### 1. `int x = -1` 在 32 位机器上的十六进制表示是什么？为什么？

**核心结论：** 32 位补码中，`-1` 的位模式是 `0xFFFFFFFF`。

**为什么：** 补码表示负数的规则是“按位取反再加 1”。`+1` 是：

```text
00000000 00000000 00000000 00000001
```

取反得到全 1 末位为 0，再加 1，结果就是 32 个 1：

```text
11111111 11111111 11111111 11111111
```

**记忆抓手：** 在补码里，`-1` 永远是“全 1”；`0` 永远是“全 0”。

### 2. `unsigned int a = 0; a - 1` 的结果是多少？说明什么性质？

**核心结论：** 在 32 位 `unsigned int` 中，结果是 `4294967295`，也就是 `UINT_MAX` 或 `0xFFFFFFFF`。

**为什么：** 无符号整数运算按模 `2^w` 进行。32 位时就是模 `2^32`：

```text
0 - 1 ≡ 2^32 - 1  (mod 2^32)
```

所以不会得到数学意义上的 `-1`，而是绕回最大无符号数。

**易错点：** 这通常叫无符号整数的“回绕”。C 标准定义了无符号回绕行为；有符号整数溢出则是未定义行为。

### 3. 为什么 `(int)(-1) > (unsigned int)(0)` 的结果是 true？

**核心结论：** 比较时 `-1` 会被转换成无符号数，变成 `UINT_MAX`，所以大于 `0`。

**底层过程：**

```c
(int)(-1) > (unsigned int)(0)
```

两边类型不同，C 的 usual arithmetic conversions 会把有符号 `int` 转成 `unsigned int`。32 位下：

```text
(unsigned int)-1 = 0xFFFFFFFF = 4294967295
```

于是实际比较变成：

```text
4294967295 > 0
```

结果为 true。

**记忆抓手：** 有符号数和无符号数混算时，要先问：“有符号数是不是被迫穿上了 unsigned 的外衣？”

### 4. `float` 只有 32 位，却能表示比 `int` 大得多的数，代价是什么？

**核心结论：** `float` 用“科学计数法式”的编码扩大范围，代价是不能精确表示所有整数和小数。

**为什么：** 32 位 `float` 被分成：

```text
sign 1 位 + exponent 8 位 + fraction 23 位
```

指数位负责范围，尾数位负责精度。指数让 `float` 能表示很大的数，但尾数位有限，所以大数附近的间隔会变大。

**典型例子：**

```c
float x = 16777216.0f;   // 2^24
float y = x + 1.0f;
```

此时 `y` 仍然等于 `16777216.0f`，因为 32 位 `float` 已经没有足够尾数位区分 `2^24` 和 `2^24 + 1`。

**记忆抓手：** `int` 是“范围较小但整数精确”；`float` 是“范围很大但间隔不均匀”。

### 5. 为什么 `+0 == -0` 为 true，而 `NaN != NaN` 为 true？

**核心结论：** `+0` 和 `-0` 位模式不同但数值语义相等；`NaN` 表示“不是一个可比较的数”，所以它和任何值都不相等，包括自己。

**为什么有 `+0` 和 `-0`：** IEEE 754 保留符号位，零也可以带符号。这有助于表达方向信息，例如 `1.0 / +0 = +inf`，`1.0 / -0 = -inf`。

**为什么 `NaN != NaN`：** `NaN` 来自无效运算，例如 `0.0/0.0`、`sqrt(-1.0)`。标准规定只要比较中有 `NaN`，普通相等比较就为 false，所以：

```c
NaN == NaN   // false
NaN != NaN   // true
```

**易错点：** 检测 NaN 应使用 `isnan(x)`，不要依赖普通大小比较。

### 6. `a[0]` 地址是 `0x100`，`a[2]` 地址是多少？`*(a+3)` 和 `a[3]` 有什么区别？

**核心结论：** 若 `int` 占 4 字节，`a[2]` 地址是 `0x108`。`*(a+3)` 和 `a[3]` 在语义上等价。

**地址计算：**

```text
&a[i] = &a[0] + i * sizeof(int)
&a[2] = 0x100 + 2 * 4 = 0x108
```

**为什么等价：** C 语言规定：

```c
a[i] == *(a + i)
```

所以 `a[3]` 就是 `*(a+3)`。

**易错点：** `a + 3` 不是地址加 3 个字节，而是加 3 个元素，即加 `3 * sizeof(*a)` 字节。

### 7. 大端序和小端序的区别是什么？`0x01020304` 在小端机器上如何存？

**核心结论：** 大端把高位字节放低地址；小端把低位字节放低地址。小端机器中，从低地址到高地址是：

```text
04 03 02 01
```

**解释：** 对整数 `0x01020304` 来说，四个字节从高到低分别是：

```text
01 02 03 04
```

小端序强调“低有效字节先放”，所以内存从低地址开始看到的是 `04`。

**记忆抓手：** 小端的“小”，可以记成“小权重字节在前”。

### 8. 左移 `x << k` 等价于乘以 `2^k`，什么时候会失效？

**核心结论：** 只有在位数足够、没有溢出、移位合法时，左移才可看作乘以 `2^k`。

**主要失效情况：**

1. **有符号溢出：** 对 signed int，若结果超出可表示范围，行为未定义。
2. **负数左移：** C 中对负的有符号整数左移通常属于未定义或不可移植行为。
3. **移位位数非法：** `k < 0` 或 `k >=` 类型位宽，行为未定义。
4. **无符号回绕：** 对 unsigned，结果按模 `2^w` 回绕，数学乘法结果可能已经丢高位。

**记忆抓手：** 左移是位操作，不是数学乘法；把它当乘法前，要先检查“溢出”和“类型”。

### 9. 为什么比较两个 `float` 通常用 `fabs(a-b)<eps`？

**核心结论：** 浮点数经过运算后会产生舍入误差，数学上相等的结果，机器中位模式可能不完全相等。

**典型例子：**

```c
0.1 + 0.2 == 0.3   // 通常为 false
```

因为 `0.1`、`0.2`、`0.3` 在二进制浮点中都不能精确表示。

**常见写法：**

```c
fabs(a - b) < eps
```

这表示“二者足够接近”。对于很大或很小的数，还应考虑相对误差，例如比较 `fabs(a-b) <= eps * max(fabs(a), fabs(b))`。

**易错点：** `eps` 不是越小越好，要和数据规模、计算误差来源匹配。

### 10. `sizeof(int)` 和 `sizeof(int*)` 在 32/64 位系统上是多少？

**核心结论：** 常见平台上 `sizeof(int)` 在 32 位和 64 位系统通常都是 4；`sizeof(int*)` 在 32 位系统是 4，在 64 位系统是 8。

**为什么：** `int` 大小由 ABI 和编译器约定决定，不必随机器位数变化。指针保存的是地址，地址空间越大，指针通常越宽。

**记忆抓手：**

```text
int 大小看 ABI；
pointer 大小看地址空间。
```

### 11. 函数调用时栈帧里通常保存什么？谁负责保存哪些寄存器？

**核心结论：** 栈帧保存返回地址、旧帧指针、局部变量、临时溢出值、部分参数和需要保护的寄存器。寄存器分为 caller-saved 和 callee-saved。

**典型栈帧内容：**

```text
返回地址
旧的 rbp
局部变量
寄存器保存区
临时 spill 空间
超过寄存器传参能力的参数
```

**x86-64 SysV 常见规则：**

```text
caller-saved: rax, rcx, rdx, rsi, rdi, r8-r11
callee-saved: rbx, rbp, r12-r15
```

**理解方法：** caller-saved 表示“调用别人前你自己保存”；callee-saved 表示“被调用者用了就要恢复”。

### 12. `%rsp` 和 `%rbp` 是什么？函数调用生命周期中如何变化？

**核心结论：** `%rsp` 指向当前栈顶；`%rbp` 常作为当前栈帧的固定基准点。

**典型过程：**

```asm
call f          ; 压入返回地址，跳到 f
push %rbp       ; 保存旧 rbp
mov %rsp,%rbp   ; 建立新栈帧
sub $N,%rsp     ; 给局部变量分配空间
...
leave           ; 等价于 mov %rbp,%rsp; pop %rbp
ret             ; 弹出返回地址并跳回
```

**二者区别：** `%rsp` 会随着 `push`、`pop`、局部变量分配而变化；`%rbp` 在函数体内通常保持稳定，便于通过固定偏移访问参数和局部变量。

**易错点：** 优化编译时可能省略 `%rbp`，直接用 `%rsp` 或寄存器管理栈帧。

### 13. signed 和 unsigned 的 `if (a > b)` 汇编有什么不同？

**核心结论：** `cmp` 指令可能一样，但后续条件跳转不同：有符号比较用 signed 条件码，无符号比较用 unsigned 条件码。

**示意：**

```asm
cmpl %esi, %edi
jg   label     ; signed: greater
ja   label     ; unsigned: above
```

**为什么：** CPU 的同一次减法会设置多个标志位。signed 比较关注符号溢出相关标志，unsigned 比较关注进位/借位相关标志。

**展开理解：** `cmp a, b` 本质上做的是 `b - a`，但不保存结果，只设置条件码。之后到底用哪种“解释方式”，取决于跳转指令：

```text
有符号比较：看 SF、OF、ZF 等标志，关心结果的正负和有符号溢出
无符号比较：看 CF、ZF 等标志，关心是否发生借位
```

例如 `-1` 的位模式是 `0xFFFFFFFF`。如果按 signed 看，它是 `-1`；如果按 unsigned 看，它是 `4294967295`。同一串位，解释规则不同，比较结论就可能相反。

**记忆抓手：**

```text
signed:   jl / jle / jg / jge
unsigned: jb / jbe / ja / jae
```

### 14. 缓冲区溢出攻击的原理是什么？覆盖的是栈上哪里？

**核心结论：** 攻击者向栈上缓冲区写入超过边界的数据，覆盖相邻栈内容，最典型目标是返回地址。

**栈上大致布局：**

```text
高地址
返回地址
保存的 rbp
局部变量 buf[]
低地址
```

如果 `buf` 写越界，可能覆盖保存的 `rbp`、返回地址或其他控制数据。函数 `ret` 时会从栈中取返回地址，若该地址被篡改，控制流就可能被劫持。

**防护机制：** 栈金丝雀、ASLR、NX、PIE、边界检查函数等都会提高攻击难度。

<a id="q15"></a>

### 15. `lea` 和 `mov` 的本质区别是什么？`leaq 8(%rdi,%rsi,4),%rax` 算什么？

**核心结论：** `mov` 访问数据；`lea` 只计算地址表达式，不访问内存。

**指令含义：**

```asm
leaq 8(%rdi,%rsi,4), %rax
```

计算：

```text
rax = rdi + rsi * 4 + 8
```

它不会去这个地址取值，只是把这个算术结果放进 `%rax`。

**记忆抓手：** `lea` 名字叫 load effective address，但常被编译器当作“带比例的加法器”。

<a id="q16"></a>

### 16. 递归函数在汇编层面如何实现？性能开销来自哪里？

**核心结论：** 递归本质上是函数反复调用自己，每一层递归都有独立栈帧。

**汇编层面的关键动作：**

```text
判断 base case
保存当前层必要状态
call 自己
返回后继续计算
ret
```

**主要开销：**

1. 每层都有 `call/ret` 控制转移开销。
2. 每层都要建立和销毁栈帧。
3. 深递归可能造成栈空间压力。
4. 相比循环，递归常让编译器更难优化。

**易错点：** 尾递归在某些情况下可被优化成循环，但 C 编译器不保证一定做。

<a id="q17"></a>

### 17. `if-else` 链和 `switch` 在汇编层面有什么不同？

**核心结论：** `if-else` 常编译成一串比较和条件跳转；密集整数 `switch` 可能编译成跳转表。

**if-else 链：**

```text
cmp x, 1
je case1
cmp x, 2
je case2
...
```

**switch 跳转表：**

```text
检查 x 是否在范围内
用 x 作为索引查表
间接跳转到目标代码
```

**什么时候用跳转表：** case 值比较密集、范围不大时更划算。若 case 很稀疏，编译器可能仍使用比较链或二叉搜索。

### 18. 函数指针的底层实质是什么？

**核心结论：** 函数指针保存的是代码区中某个函数入口地址。

**调用方式：**

```c
int (*fp)(int) = f;
fp(3);
```

底层相当于“从函数指针中取出地址，然后做间接调用”：

```asm
call *%rax
```

**易错点：** 函数指针指向代码，不指向普通数据；它让控制流目标在运行时决定，因此也会影响编译器内联和优化。

### 19. 为什么局部变量在函数返回后不能访问？返回局部变量指针会怎样？

**核心结论：** 局部变量通常位于当前函数栈帧中；函数返回后栈帧失效，指向它的指针变成悬空指针。

**例子：**

```c
int *f() {
    int x = 10;
    return &x;   // 错误
}
```

`f` 返回后，`x` 所在的栈空间可能被后续函数调用复用。此时访问返回的指针属于未定义行为：可能看似能读到旧值，也可能读到垃圾值或崩溃。

**安全做法：** 返回值本身、使用调用者传入的缓冲区、动态分配内存，或使用生命周期更长的对象。

<a id="q20"></a>

### 20. 为什么缓冲区溢出有时 segfault，有时却“悄悄成功”？

**核心结论：** 是否崩溃取决于越界写到了哪里，以及那个地址是否映射、是否有权限、是否破坏了关键控制数据。

**会 segfault 的情况：**

1. 写到未映射虚拟页。
2. 写到只读页。
3. 覆盖返回地址后跳到非法地址。
4. 触发栈保护机制。

**可能“悄悄成功”的情况：**

1. 越界仍落在当前栈页内。
2. 只覆盖了暂时没被使用的局部变量或 padding。
3. 覆盖内容没有立刻影响控制流。

**跨章节串联：** 数组连续性解释“为什么能越界写到旁边”；栈布局解释“旁边是什么”；虚拟内存解释“为什么有时访问非法地址会 fault”。

### 21. `.o` 和可执行文件都是 ELF，它们有什么不同？

**核心结论：** `.o` 是可重定位文件，里面还有未最终确定的地址和重定位信息；可执行文件已经完成主要链接工作，可以被加载运行。

**`.o` 文件：**

```text
.text、.data、.bss、.rodata
.symtab 符号表
.rel.text / .rela.text 重定位表
未解析或未重定位的外部引用
```

**可执行文件：**

```text
合并后的代码和数据
程序入口 e_entry
Program Header
已修正的大部分地址引用
动态链接信息（若是动态链接）
```

**再深一层：节和段的区别。** `.o` 文件更关心“节 section”，因为链接器要把各个 `.o` 的 `.text`、`.data`、`.bss` 等节合并。可执行文件更关心“段 segment”，因为加载器要按照 Program Header 把若干节组合成可装入内存的段。

```text
链接器看 Section Header：方便链接、合并、重定位
加载器看 Program Header：方便 mmap 到虚拟地址空间
```

所以答题时可以说：`.o` 仍然带有重定位表和未决符号，服务“链接”；可执行文件已经组织成可加载的内存映像，服务“运行”。

**记忆抓手：** `.o` 是“零件”；可执行文件是“能装载进进程的成品”。

<a id="q22"></a>

### 22. 符号解析和重定位分别解决什么问题？

**核心结论：** 符号解析回答“这个名字到底指谁”；重定位回答“它最终在内存中的地址是多少，引用处该填什么”。

**符号解析：** 将每个符号引用绑定到一个符号定义。例如 `printf` 来自库，`global` 来自某个 `.o`。

**重定位：** 当各个节被合并并分配最终地址后，链接器修改代码或数据里的地址字段，让引用指向正确位置。

**简化流程：**

```text
多个 .o -> 符号解析 -> 分配地址 -> 修改引用位置 -> 可执行文件
```

### 23. 绝对地址重定位和相对地址重定位有什么区别？

**核心结论：** 绝对重定位填入目标的绝对地址；相对重定位填入“目标地址相对当前指令位置的偏移”。

**绝对重定位：** 

```text
填入 S + A
```

其中 `S` 是符号地址，`A` 是附加常数。

**PC 相对重定位：**

```text
填入 S + A - P
```

其中 `P` 是被修改位置的地址。

**为什么相对重定位重要：** 相对寻址更适合位置无关代码（PIC），也更适合动态链接和 ASLR。

### 24. 强符号和弱符号有什么区别？同名全局变量如何处理？

**核心结论：** 函数和已初始化全局变量通常是强符号；未初始化全局变量在传统规则中常按弱符号处理。链接器按强弱规则裁决。

**规则：**

```text
多个强符号同名：链接错误
一个强符号 + 多个弱符号：选择强符号
多个弱符号：任选一个，可能埋下 bug
```

**例子：**

```c
// a.c
int x = 1;   // 强符号

// b.c
int x;       // 弱符号或 tentative definition
```

链接时通常选择 `a.c` 中的强符号。

**易错点：** 现代 GCC 默认 `-fno-common` 后，多个 tentative definition 也可能直接报 multiple definition。考试答题要按课程讲过的链接规则说明。

<a id="q25"></a>

### 25. 静态链接和动态链接有什么区别？

**核心结论：** 静态链接把库代码复制进可执行文件；动态链接在运行时或加载时把共享库映射进进程。

**静态链接：**

```text
优点：部署依赖少
缺点：文件大，多个程序重复占空间，库更新需重新链接
```

静态库还有顺序问题：链接器按命令行从左到右扫描，库通常应放在引用它的目标文件之后。

**动态链接：**

```text
优点：多个进程可共享代码页，库可单独更新
缺点：运行依赖共享库，加载和符号解析更复杂
```

`.so/.dll` 通常在程序加载时由动态链接器装入，也可以通过 `dlopen` 等机制运行时装入。

**考场展开点：**

```text
静态库 .a：本质是一组 .o 的归档，链接时只取用被引用到的目标文件
动态库 .so：运行时映射进进程地址空间，代码页可被多个进程共享
```

静态链接的“顺序陷阱”来自链接器的一遍扫描：如果库放在前面，而引用它的目标文件放在后面，扫描库时还不知道需要哪些符号，可能就不会把相应目标文件取出来。因此常写成：

```bash
gcc main.o compute.o -lm
```

而不是把 `-lm` 放在最前面。

<a id="q26"></a>

### 26. 动态库被多个进程共享时，虚拟地址和物理页面如何对应？

**核心结论：** 多个进程可以把同一个动态库的只读代码页映射到相同的物理页；各进程看到的虚拟地址不一定相同。

**典型情况：**

```text
进程 A 的 VA -> 物理页 P
进程 B 的 VA -> 同一个物理页 P
```

共享最多的是只读代码段和只读数据段。可写数据段通常不能直接共享，否则进程会互相修改；它们常通过私有映射或写时复制（COW）获得各自副本。

**易错点：** “虚拟地址相同”和“物理页相同”是两件事。共享的本质是页表项指向同一物理页。

### 27. 时间局部性和空间局部性是什么？行遍历和列遍历哪个好？

**核心结论：** 时间局部性是“刚访问过的还会再访问”；空间局部性是“访问了一个地址，附近地址也可能被访问”。C 的二维数组按行优先存储，所以按行遍历局部性更好。

**对比：**

```c
// A: 按行
for (i = 0; i < N; i++)
    for (j = 0; j < N; j++)
        sum += a[i][j];

// B: 按列
for (j = 0; j < N; j++)
    for (i = 0; i < N; i++)
        sum += a[i][j];
```

按行遍历访问的是连续内存，Cache 一次取入一个块后，后续元素很可能命中。按列遍历跨度大，容易浪费 Cache 块。

<a id="q28"></a>

### 28. 存储层次金字塔是什么？为什么能协同工作？

**核心结论：** 存储层次从快到慢、从小到大大致是：寄存器、L1/L2/L3 Cache、主存、SSD/HDD、远程存储。

**为什么需要分层：** 快存储贵且容量小，慢存储便宜且容量大。系统用少量快速存储缓存大量慢速存储中的热点数据。

**为什么能工作：** 程序具有局部性。只要最近或附近的数据很可能再次被访问，小而快的缓存就能显著降低平均访问时间。

**记忆抓手：**

```text
越靠近 CPU：越快、越小、越贵；
越远离 CPU：越慢、越大、越便宜。
```

<a id="q29"></a>

### 29. 矩阵乘法 `i-j-k` 和 `i-k-j` 哪个更尊重 Cache？

**核心结论：** 对 C 语言行优先数组，通常 `i-k-j` 更好。

**`i-j-k` 的问题：**

```c
for (i)
  for (j)
    for (k)
      C[i][j] += A[i][k] * B[k][j];
```

内层 `k` 变化时，`A[i][k]` 连续，但 `B[k][j]` 按列访问，跨度大，Cache 局部性差。

**`i-k-j` 的优势：**

```c
for (i)
  for (k)
    for (j)
      C[i][j] += A[i][k] * B[k][j];
```

内层 `j` 变化时，`B[k][j]` 连续，`C[i][j]` 也连续；`A[i][k]` 被固定后可重复使用。

**记忆抓手：** C 数组行优先，最内层循环最好沿着行走。

<a id="q30"></a>

### 30. 加载完成了什么？为什么加载前必须有新进程？

**核心结论：** 加载是把可执行文件映射成一个进程的虚拟地址空间，并设置初始运行状态。进程是运行程序的容器，所以必须先有进程，再把程序装进去。

**`execve` 主要做的事：**

```text
读取 ELF 头和 Program Header
映射代码段、数据段、动态库
创建栈，放入 argv/envp
设置堆和 .bss
设置入口地址
丢弃旧程序的用户态地址空间
```

**为什么常见是 fork + execve：** shell 先 `fork` 出子进程，子进程再 `execve` 新程序。这样 shell 自己不会被新程序替换，还能继续等待、管理和接收命令。

**关键区别：**

```text
fork：复制当前进程，产生一个新的进程身份
execve：不产生新进程，只替换当前进程正在运行的程序
```

因此 `fork` 之后，父子进程的 PID 不同；`execve` 之后，子进程的 PID 通常不变，但用户态地址空间换成了新程序。这个点很适合和虚拟内存联系起来：`execve` 改的是页表映射和进程的用户态内存布局。

<a id="q31"></a>

### 31. Cache 由哪三个参数决定？地址如何分解？

**核心结论：** Cache 组织由 `S`、`E`、`B` 决定：`S` 是组数，`E` 是每组行数，`B` 是块大小。容量 `C = S * E * B`。

**地址分解：**

```text
| tag | set index | block offset |
```

若 `B = 2^b`，则低 `b` 位是块内偏移。若 `S = 2^s`，则接下来的 `s` 位是组索引。剩下高位是 tag。

**公式化写法：**

```text
b = log2(B)
s = log2(S)
t = 地址位数 - s - b
```

例如 32 位地址、块大小 `B=64=2^6`、组数 `S=256=2^8`：

```text
block offset = 6 位
set index    = 8 位
tag          = 32 - 6 - 8 = 18 位
```

**查找流程：**

```text
用 set index 找组
在组内比较 tag
tag 匹配且 valid=1 -> 命中
否则 miss，从下级存储取块
```

<a id="q32"></a>

### 32. 直接映射、组相联、全相联 Cache 的优缺点是什么？

**核心结论：** 直接映射最快但冲突多；全相联冲突少但硬件贵；组相联是折中。

**对比：**

```text
直接映射：每个块只能去一个位置，简单快，最容易冲突
组相联：每个块可去某一组内多个位置，折中
全相联：每个块可去任意位置，冲突少，查找成本高
```

**抖动 thrashing：** 当多个频繁访问的地址不断映射到同一个 cache 位置或同一小组，彼此反复替换，就会抖动。直接映射最容易出现。

<a id="q33"></a>

### 33. 写命中和写不命中的策略有什么区别？

**核心结论：** 写命中主要有 write-through 和 write-back；写不命中主要有 write-allocate 和 no-write-allocate。

**写命中：**

```text
write-through：同时写 Cache 和下级存储，简单但写流量大
write-back：先只写 Cache，标 dirty，替换时再写回，流量小但控制复杂
```

**写不命中：**

```text
write-allocate：先把块取入 Cache，再写
no-write-allocate：不取入 Cache，直接写下级存储
```

**常见搭配：** write-back 常搭配 write-allocate；write-through 可搭配 no-write-allocate。

<a id="q34"></a>

### 34. 为什么 Cache 块大小不是越大越好？矩阵分块大小如何选？

**核心结论：** 块太小浪费空间局部性；块太大增加 miss penalty，也容易把有用数据挤出去。

**块大小权衡：**

```text
块大：一次带来更多相邻数据，空间局部性好
块太大：传输慢，污染 Cache，可容纳块数变少
块小：替换灵活
块太小：无法充分利用连续访问
```

**矩阵分块：** 分块的目的不是让单个块等于 Cache 大小，而是让计算中同时活跃的几个小矩阵块能一起放进 L1/L2 Cache。

例如普通矩阵乘法常要同时用到 A、B、C 的块，可粗略要求：

```text
3 * block_size^2 * sizeof(element) < 可用 Cache 容量
```

<a id="q35"></a>

### 35. Cache 和 TLB 的相似点、不同点是什么？地址翻译和 Cache 查找顺序如何？

**核心结论：** Cache 缓存数据，TLB 缓存地址翻译。二者都用小而快的硬件结构加速常见访问。

**对比：**

```text
TLB:   VPN -> PPN，服务虚拟地址翻译
Cache: 地址 -> 数据，服务数据访问
```

**普通理解顺序：**

```text
虚拟地址 -> TLB/页表翻译 -> 物理地址 -> Cache 查找 -> 主存
```

**实际优化：** 很多 L1 Cache 使用 VIPT（Virtually Indexed, Physically Tagged），可以用页内偏移中的 index 位先查 Cache，同时 TLB 做翻译，最后用物理 tag 验证。

**为什么 VIPT 可行：** 页内偏移在地址翻译前后不变。虚拟地址和物理地址的页号会变，但页内偏移相同；只要 L1 Cache 的组索引位落在页内偏移范围内，就可以先用虚拟地址的这些位选组，等 TLB 给出物理页号后再比较物理 tag。

**容易混淆的两条路径：**

```text
地址翻译路径：VA -> TLB/page table -> PA
数据访问路径：PA -> Cache -> memory
```

现代处理器会把两条路径部分并行做，但逻辑上仍然要得到正确物理地址和权限检查结果，不能绕过虚拟内存保护。

**易错点：** TLB miss 不等于 Cache miss；一个是翻译没缓存，一个是数据没缓存。

### 36. 虚拟内存解决哪三个核心问题？

**核心结论：** 虚拟内存主要解决：地址空间抽象、内存保护、把主存当作磁盘的缓存。

**三个作用：**

```text
抽象：每个进程以为自己独占连续地址空间
保护：进程之间隔离，页级权限控制读/写/执行
缓存：只把需要的页放入物理内存，不够时换出到磁盘
```

**如果没有虚拟内存：** 程序员或系统必须手动管理物理地址、避免程序互相覆盖、自己处理装入和换出，这会非常困难且不安全。

<a id="q37"></a>

### 37. 页表是什么？TLB 不命中时一次内存访问需要几次物理内存访问？

**核心结论：** 页表记录虚拟页到物理页的映射和权限。TLB 不命中时，需要先访问页表得到物理地址，再访问真正数据。

**单级页表简化情况：**

```text
1 次访问页表项
1 次访问目标数据
合计至少 2 次物理内存访问
```

**多级页表：** 若页表有 n 级，最坏需要访问 n 个页表项，再访问数据，所以是 `n + 1` 次。不过页表项本身也可能被 Cache 缓存。

**具体一点：** 以常见 4KB 页为例，虚拟地址低 12 位是页内偏移，不参与页表查找；高位虚拟页号被拆成多级索引。若是四级页表，TLB 不命中且页表项都不在 Cache 中，理论上可能需要：

```text
4 次访问各级页表项 + 1 次访问真实数据 = 5 次内存访问
```

实际机器上，页表项也会走 Cache，所以真实代价不一定等于 5 次主存访问，但“TLB miss 会额外查页表”这个逻辑不变。

**多级页表解决的问题：** 避免为整个巨大虚拟地址空间分配一张完整连续页表，只为实际使用的地址范围分配页表页。

### 38. 缺页发生时，操作系统做了什么？

**核心结论：** 缺页是一次 fault。CPU 转入内核，操作系统判断访问是否合法；合法则把页调入或创建，更新页表后重新执行出错指令。

**流程：**

```text
访问虚拟地址
页表项无效或权限不满足
触发 page fault
进入内核 fault handler
检查地址是否合法
若非法：发送 SIGSEGV
若合法：分配物理页或从磁盘读入
更新页表和 TLB
返回用户态，重试原指令
```

**常见合法缺页：**

```text
按需调页：代码或数据页还没真正读入内存
栈增长：访问到栈附近的合法扩展区域
写时复制：fork 后首次写共享只读页，需要复制一份
文件映射：mmap 的文件页首次被访问
```

**非法缺页：** 地址根本不属于进程的合法区域，或者权限不对，例如向只读代码段写入。此时内核不能“修好”它，通常给进程发送 `SIGSEGV`。

**易错点：** 缺页不一定是错误；访问尚未装入的合法页面也会缺页。

<a id="q39"></a>

### 39. 虚地址 `0x03D4` 发出后，经过 TLB、Cache、主存的过程是什么？

**核心结论：** 先把虚拟地址翻译成物理地址，再用物理地址访问 Cache；若 Cache miss，再访问主存。

**通用步骤：**

```text
1. CPU 产生虚拟地址 VA = 0x03D4
2. 将 VA 分成 VPN 和 VPO
3. 用 VPN 查 TLB
4. TLB 命中：得到 PPN
5. TLB 不命中：查页表；若页不存在，触发缺页
6. PPN + VPO 组成物理地址 PA
7. 用 PA 查 L1 Cache
8. Cache 命中：返回数据
9. Cache 不命中：从主存取块，填入 Cache，再返回数据
```

**记忆抓手：** TLB 管“地址能不能翻译快一点”；Cache 管“数据能不能取到快一点”。

<a id="q40"></a>

### 40. 两个进程访问同一虚拟地址 `0x400000`，是同一物理内存吗？

**核心结论：** 通常不是。不同进程有不同页表，同一个虚拟地址可以映射到不同物理页。

**普通隔离：**

```text
进程 A: VA 0x400000 -> PPN 1
进程 B: VA 0x400000 -> PPN 9
```

它们互不影响。

**共享内存：**

```text
进程 A: VA 0x400000 -> PPN 5
进程 B: VA 0x700000 -> PPN 5
```

共享的关键不是虚拟地址是否相同，而是页表项是否指向同一个物理页。

### 41. 异常、中断、陷阱、故障有什么区别？缺页属于哪种？

**核心结论：** 异常是控制流突变的总称；中断来自外部设备；陷阱是有意触发的同步异常；故障是可能被修复并重试的同步异常。缺页属于故障。

**对比：**

```text
中断 interrupt：异步，来自 I/O、时钟等外部事件
陷阱 trap：同步、有意触发，如系统调用
故障 fault：同步、可能修复，如缺页
终止 abort：严重错误，通常不可恢复
```

**缺页为什么是 fault：** 触发缺页的那条指令还没有完成。操作系统把页准备好后，可以重新执行同一条指令。

**更清楚的分类方式：**

```text
异步/同步：
  中断是异步的，和当前指令没有直接因果关系。
  trap、fault、abort 是同步的，由当前指令触发。

是否有意：
  trap 通常是程序有意触发，例如 syscall。
  fault 通常不是程序主动想进入内核，而是执行中遇到可修复问题。

是否可恢复：
  fault 可能修复后重试。
  abort 通常不可恢复，例如严重硬件错误。
```

答这题时，不要把“异常”和“中断”并列成完全同级概念；更准确地说，异常控制流是大类，中断、陷阱、故障、终止是具体类型。

### 42. 进程和程序有什么区别？`fork()` 后地址空间是什么关系？

**核心结论：** 程序是静态文件；进程是程序运行起来后的实例。`fork()` 后父子进程逻辑上有相同地址空间，但实际通过写时复制共享物理页。

**fork 后：**

```text
父子看到的变量初值相同
页表最初可指向同一物理页
某一方写入时触发 COW，复制出私有页
```

**所以：** 父子进程修改“同名变量”通常不会互相影响，因为写入后它们映射到不同物理页。

### 43. 僵死进程怎么产生？为什么要 `wait()`？

**核心结论：** 子进程退出后，父进程还没有回收它的退出状态时，子进程处于 zombie 状态。

**为什么要 `wait/waitpid`：**

```text
回收进程表项
取得子进程退出状态
实现父子进程同步
```

**父进程先退出怎么办：** 子进程会被 `init` 或 `systemd` 等进程收养。它退出后，收养它的进程负责回收。

**易错点：** zombie 不是还在运行的进程，而是“已经结束但还有记录没被父进程取走”。

### 44. 信号的发送、接收、处理分别发生什么？“不排队”是什么意思？

**核心结论：** 发送信号是设置目标进程的 pending 标记；接收信号是内核在合适时机把未阻塞信号交给进程；处理信号是执行默认动作、忽略或调用用户处理函数。

**生命周期：**

```text
send: kill/alarm/键盘/内核异常等产生信号
pending: 信号被标记为待处理
blocked: 若被阻塞，暂不递送
delivery: 进程回到用户态前处理
handler: 执行处理函数或默认动作
```

**不排队：** 对传统信号，同一种信号在 pending 期间只保留一个标记。连续来了多个同类信号，可能合并成一次处理。

**易错点：** 实时信号可以排队；课程里常说“不排队”通常指传统 Unix signals。

**为什么 `SIGCHLD` 特别容易考：** 多个子进程几乎同时退出时，父进程可能只收到一次 `SIGCHLD` 处理机会。如果 handler 里只调用一次 `wait()`，就可能只回收一个子进程，剩下的子进程继续保持 zombie。

更稳妥的思路是循环回收：

```c
while (waitpid(-1, &status, WNOHANG) > 0) {
    /* reap one child */
}
```

**信号处理函数的坑：** handler 会打断主程序执行，因此里面应尽量少做事。严格来说，`printf` 这类函数不是 async-signal-safe，真实工程中常在 handler 里只设置一个标志位，主循环再处理。

### 45. `fork()+execve()` 如何实现运行新程序？

**核心结论：** `fork` 创建子进程，`execve` 在子进程中用新程序替换原地址空间。

**以 shell 运行命令为例：**

```text
shell 调用 fork
父进程继续作为 shell，可 wait 子进程
子进程调用 execve("/bin/ls", ...)
内核丢弃子进程旧的代码、数据、堆、栈映射
装入 /bin/ls 的 ELF 映射
从新程序入口开始运行
```

**关键点：** `execve` 不创建新进程；它替换当前进程的用户态地址空间。PID 通常保持不变。

### 46. 什么是关键路径？为什么单纯循环展开不能提升浮点累加性能？

**核心结论：** 关键路径是计算中最长的依赖链。浮点累加有循环携带依赖，单纯展开没有打破依赖；多路并行累加能缩短单条依赖链。

**普通累加：**

```c
sum = sum + a[i];
```

下一次加法必须等上一次 `sum` 出来，形成串行依赖链。

**单纯展开：**

```c
sum += a[i];
sum += a[i+1];
sum += a[i+2];
```

仍然都依赖同一个 `sum`。

**多路并行：**

```c
sum0 += a[i];
sum1 += a[i+1];
sum2 += a[i+2];
sum3 += a[i+3];
```

多个累加器形成多条独立依赖链，CPU 可以并行发射和执行更多指令。

**延迟 vs 吞吐量：** 单个浮点加法可能有几拍延迟，但处理器通常可以每拍启动一个新的浮点加法。若只有一个 `sum`，下一次加法必须等上一次完成，受“延迟”限制；若有多个独立累加器，处理器可以连续启动多条加法，接近“吞吐量”上限。

**最后一步：** 多路累加不是不要总和，而是先分别累加：

```c
sum0, sum1, sum2, sum3
```

循环结束后再做：

```c
sum = sum0 + sum1 + sum2 + sum3;
```

这样大部分循环时间都在多条短依赖链上运行，而不是一条很长的依赖链上排队。

### 47. 什么是内存别名？为什么会阻止优化？

**核心结论：** 内存别名是两个指针可能指向同一块内存。若编译器不能证明它们不同，就不能随意重排、缓存或合并内存访问。

**例子：**

```c
void f(int *p, int *q) {
    *p = 1;
    *q = 2;
    printf("%d", *p);
}
```

如果 `p == q`，最后打印的是 `2`；如果 `p != q`，打印的是 `1`。编译器不知道二者是否别名，就必须保守处理。

**解决思路：** 使用局部变量保存中间值，或在符合条件时使用 `restrict` 告诉编译器指针不别名。

### 48. 为什么全局变量会阻碍编译器优化？

**核心结论：** 全局变量可能被其他函数、其他文件甚至信号处理函数修改，编译器必须保守地重新读取和写回。

**问题来源：**

```c
extern int g;

void f() {
    g++;
    h();
    g++;
}
```

调用 `h()` 后，编译器通常不能假设 `g` 没变，因为 `h` 可能访问或修改全局变量。

**与局部变量对比：** 局部变量若地址没有逃逸，编译器更容易把它放在寄存器中，做常量传播、公共子表达式消除和循环优化。

**实践建议：** 热路径中少用可变全局状态；必要时把全局值读入局部变量，计算完再写回。

### 49. Amdahl 定律：把占 10% 时间的模块优化到 0，整体最多快多少？

**核心结论：** 最多加速到 `1 / 0.9 = 1.111...` 倍，也就是整体最多快约 11.1%。

**公式：**

```text
speedup = 1 / ((1 - alpha) + alpha / k)
```

其中 `alpha` 是被优化部分原本占比，`k` 是这部分的加速倍数。若优化到零耗时，相当于 `k -> infinity`：

```text
speedup = 1 / (1 - 0.1) = 1 / 0.9 ≈ 1.11
```

**启示：** 优化要先找热点。把 1% 的代码优化 100 倍，整体收益也很有限；真正值得优化的是高占比瓶颈。

### 50. 大数组累加同时受 Cache miss、串行依赖、TLB miss、页错误影响，如何排序？

**核心结论：** 严格说要看“单次代价 × 发生次数”。若按单次事件代价排序，通常是：

```text
页错误 > TLB miss > Cache miss > 关键路径串行依赖
```

**逐项理解：**

```text
页错误：可能涉及内核、磁盘 I/O、页表更新，代价最大
TLB miss：需要页表遍历，可能访问多级页表
Cache miss：需要访问下级 Cache 或主存
串行依赖：每次只是限制流水线并行度，单次代价相对小
```

**但考试答题要补一句：** 对“已经在内存中的大数组顺序累加”，页错误通常只在首次触碰页面或内存不足时出现，稳态性能更常由 Cache miss、TLB miss 和累加依赖共同决定。若数据流式访问且工作集很大，Cache miss 频率可能远高于 TLB miss；若每页访问很少，TLB miss 影响会更明显。

**更成熟的答法：** 这题如果问“对性能影响从大到小”，不要只给死排名，先说明评价口径。

```text
按单次代价：页错误 > TLB miss > Cache miss > 串行依赖
按稳定运行中的总影响：取决于发生频率，Cache miss 和串行依赖常常更主导
```

例如顺序累加大数组时，每个元素都有累加依赖；Cache miss 大约按 cache block 粒度发生；TLB miss 大约按页粒度发生；页错误通常只在页面首次装入或内存压力很大时发生。于是“页错误单次最贵”不等于“页错误一定是总时间最大来源”。

**对应章节：**

```text
页错误：虚拟内存 + 异常控制流
TLB miss：虚拟内存
Cache miss：高速缓存 / 存储层次
关键路径串行依赖：程序性能优化
```

**记忆抓手：** 不要只背排名；性能判断永远看“每次多贵”和“发生几次”。
