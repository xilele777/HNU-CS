# 题目解析

## （1）与待链接的可重定位目标文件相比，链接后的可执行文件目标文件的内容有哪些变化？为什么会有这些变化？

**变化：**

1. **ELF 头中 `e_type` 字段改变**：从 `ET_REL`（可重定位）变为 `ET_EXEC`（可执行），并设置了程序入口地址 `e_entry`。

2. **增加了程序头部表（Program Header Table）**：可重定位文件只有节头部表，可执行文件增加了程序头部表，用于描述段（Segment）到内存的映射关系（虚拟地址、文件偏移、访问权限等）。

3. **`.rel.text` 和 `.rel.data` 重定位节被移除**：链接后所有重定位条目已被解析，这些节不再需要。

4. **多个目标文件的同名节被合并**：多个 `.o` 文件中的 `.text`、`.data`、`.bss` 等节分别合并为单一的节，所有符号引用已解析。

5. **地址绑定完成**：可重定位文件中地址从 0 开始（相对地址），链接后所有符号地址被绑定为绝对虚拟地址，节和符号都分配了确定的内存地址。

6. **`.symtab` 符号表可能被精简或去除**：链接完成后符号表不再必需（除非保留用于调试），可执行文件中的符号表可以被 strip。

7. **调试节可能被移除**：`.debug`、`.line` 等调试信息在可执行文件中是可选的，通常会被剥离。

**原因：**

可重定位目标文件是给链接器使用的中间产物，含有未解析的符号引用和相对地址，需要重定位信息。链接器的作用是将多个目标文件整合、解析符号引用、完成重定位，最终生成一个**可直接被操作系统加载器加载运行**的文件。因此可执行文件需要程序头部表来告诉加载器如何将段映射到内存（虚拟地址、权限、对齐等），而不再需要重定位信息。

## （2）如果待链接的两个目标文件分别来源于图 6.2 与图 6.3 中的两段 C 代码，可能会产生什么问题？为什么？

**问题一：变量 `y` 未定义 —— 链接错误**

在 `main11.c` 中使用了 `y = 0;`，但 `y` 从未在任何文件中声明或定义。链接器在解析符号引用时找不到符号 `y` 的定义，会报 **"undefined reference to `y`"** 错误，链接失败。

**问题二：变量 `x` 类型冲突 —— 强符号覆盖弱符号导致类型不一致**

- `main11.c` 中声明 `int x;`（未初始化全局变量），在 C 语言中属于**暂定定义（tentative definition）**，作为 COMMON 符号（弱符号），分配 4 字节。
- `f11.c` 中声明 `double x = 1.0;`（已初始化全局变量），属于**强符号**，分配 8 字节。

链接器处理规则：**强符号覆盖弱符号，多个弱符号选占用空间最大的**。因此链接器会选择 `f11.c` 中的 `double x`（8 字节），但 `main11.c` 中的代码仍按 `int`（4 字节）来读写 `x`。

导致的后果：
- `x = 15213;` 只写入 4 字节的 int 值到 8 字节的 double 变量中，高 4 字节仍是 double 1.0 的浮点表示部分。
- `printf("x = %d\n", x);` 按 `%d`（int 格式，4 字节）读取，但内存中存储的是被部分覆盖的 double 值（8 字节），输出的将是**错误/无意义的值**，而非 15213。

**问题三：函数 `f` 声明与定义不一致**

- `main11.c` 中声明 `void f(void);`（无返回值）
- `f11.c` 中定义 `int f() { ... }`（返回 int，但函数体无 `return` 语句）

虽然链接器按符号名匹配能找到 `f` 的地址，但返回值类型不匹配且在 `f11.c` 中 `f()` 函数体缺少 `return` 语句，函数执行完毕后返回值不确定，属于语义错误。

## （3）对于图 6.2 与图 6.3 的代码进行链接操作形成可执行文件 P 时，可能会发生哪些重定位？并简述这些重定位的基本过程。

假设先忽略 `y` 未定义导致的链接失败问题（或假设 `y` 已在别处定义），链接 `main11.o` 和 `f11.o` 形成可执行文件 P 时，会发生以下重定位：

### 一、重定位类型

链接时主要涉及两类重定位：

**1. 重定位 PC 相对引用（R_X86_64_PC32 / R_386_PC32）**

用于函数调用指令（如 `call` 指令）中的相对偏移量计算。这些引用位于 `.rel.text` 节中。

- `main11.o` 中对 `f()` 的 `call` 指令 —— 引用外部符号 `f`，定义在 `f11.o` 中
- `main11.o` 中对 `printf()` 的 `call` 指令 —— 引用外部符号 `printf`，定义在 libc 中

**2. 重定位绝对引用（R_X86_64_32 / R_386_32）**

用于数据访问指令中的绝对地址计算。这些引用位于 `.rel.text` 或 `.rel.data` 节中。

- `main11.o` 中对全局变量 `x` 的访问（`x = 15213;` 和 `printf` 中读取 `x`）—— 引用符号 `x`
- `f11.o` 中对全局变量 `x` 的访问（`x = 1.0;`）—— 引用符号 `x`

### 二、重定位的基本过程

**1. PC 相对引用重定位（以 `call f` 为例）**

- **链接前**：`main11.o` 中 `call f` 指令的操作数是占位值（通常为 0 或 -4），`.rel.text` 中记录了一条重定位条目，包含：
  - `offset`：需要修改的指令操作数在 `.text` 中的偏移
  - `symbol`：符号 `f`
  - `type`：`R_386_PC32`

- **链接时**：链接器合并 `main11.o` 和 `f11.o` 的 `.text` 节后，确定 `call` 指令和 `f` 函数入口的运行时地址（虚拟地址），然后计算：
  ```
  重定位值 = S + A - P
  ```
  其中：
  - `S` = 符号 `f` 的运行时地址（`f11.o` 中 `f` 的 `.text` 偏移 + 合并后基址）
  - `A` = 加数（addend），原操作数中的占位值
  - `P` = 被重定位处的运行时地址（`call` 指令下一条指令的地址）

  将计算结果写回 `call` 指令的操作数字段，使得运行时 `call` 指令能正确跳转到 `f` 的地址。

**2. 绝对引用重定位（以 `x = 15213` 为例）**

- **链接前**：`main11.o` 中访问 `x` 的 `mov` 指令操作数也是占位值，`.rel.text` 中记录了重定位条目：
  - `offset`：指令操作数偏移
  - `symbol`：符号 `x`
  - `type`：`R_386_32`

- **链接时**：链接器确定符号 `x` 的运行时地址（在 `.data` 或 `.bss` 中的绝对地址），计算：
  ```
  重定位值 = S + A
  ```
  其中：
  - `S` = 符号 `x` 的运行时地址
  - `A` = 加数

  将计算得到的 `x` 的绝对地址写回指令操作数字段。

### 三、本题涉及的具体重定位条目

| 引用位置 | 符号 | 重定位类型 | 说明 |
|---------|------|-----------|------|
| `main11.o` 中 `call f` | `f` | PC 相对 | 函数调用，`f` 定义在 `f11.o` |
| `main11.o` 中 `call printf` | `printf` | PC 相对 | 库函数调用，定义在 libc |
| `main11.o` 中 `movl $15213, x` | `x` | 绝对引用 | 写全局变量 `x` |
| `main11.o` 中 `pushl x`（printf 参数） | `x` | 绝对引用 | 读全局变量 `x` |
| `f11.o` 中 `movl $0x..., x` / `movsd ..., x` | `x` | 绝对引用 | 写全局变量 `x`（double 赋值） |

链接器通过遍历 `.rel.text` 和 `.rel.data` 中的重定位条目，逐一解析上述引用，将占位地址替换为正确的运行时地址，最终生成可执行文件 P。

## （4）请从产生新进程继而加载执行它的角度，描述在终端使用"./P"到屏幕输出相应运行结果的实际过程。

整个过程可分为以下几个阶段：

### 一、Shell 解析命令

用户在终端输入 `./P` 并按下回车后，Shell 解析该命令字符串，识别出这是一个需要执行的可执行文件路径（相对路径，当前目录下的 `P`）。

### 二、fork() 创建子进程

Shell 调用 `fork()` 系统调用，创建一个与当前 Shell 进程几乎完全相同的子进程。

- 内核为新进程分配 PID、复制父进程的 `task_struct`、页表等内核数据结构
- 子进程获得父进程虚拟地址空间的**写时复制（Copy-on-Write）**副本——页表指向相同的物理页，仅当父子任一方尝试写入时才复制物理页
- `fork()` 在父进程（Shell）中返回子进程 PID，在子进程中返回 0

### 三、execve() 加载可执行文件

子进程（此时仍是 Shell 的副本）调用 `execve("./P", argv, envp)` 系统调用，用可执行文件 P 的内容**完全替换**当前进程的地址空间。

**3.1 内核加载器的工作：**

1. **读取 ELF 头**：内核从磁盘读取 P 的 ELF 头，验证魔数（`0x7f E L F`），确认是合法的可执行文件，读取 `e_entry`（程序入口地址）。

2. **读取程序头部表**：遍历 Program Header Table，找出所有类型为 `PT_LOAD` 的段（Segment），每个段描述了一个需要映射到内存的区域。

3. **创建新地址空间**：内核回收子进程原有的地址空间，创建一个全新的虚拟地址空间。

4. **映射段到内存**：根据每个 `PT_LOAD` 段的描述（虚拟地址、文件偏移、大小、权限）：
   - 将 `.text`（代码段）以只读+执行权限（`r-x`）映射到虚拟地址空间
   - 将 `.rodata`（只读数据）以只读权限（`r--`）映射
   - 将 `.data`（已初始化数据）以读写权限（`rw-`）映射，并从文件读取初始值
   - 为 `.bss`（未初始化数据）分配匿名内存页（全零填充），以读写权限映射

5. **设置用户栈**：内核在用户空间栈顶构造初始栈，包含：
   - 命令行参数字符串（`argc`/`argv`）
   - 环境变量字符串（`envp`）
   - 辅助向量（auxiliary vector，如 AT_ENTRY、AT_PHDR 等）

6. **设置寄存器并跳转**：将 CPU 的指令指针寄存器（EIP/RIP）设置为 ELF 头中的 `e_entry`（通常是 `_start` 函数的地址），将栈指针（ESP/RSP）指向刚构造好的用户栈。然后从内核态返回用户态，控制权交给 `_start`。

### 四、程序启动与执行

**4.1 `_start` → `__libc_start_main`**

`_start`（位于 `crt0.o` 中）是 C 程序的真正入口，它负责：
- 从栈上获取 `argc`、`argv`、`envp`
- 调用 `__libc_start_main(main, argc, argv, __libc_csu_init, __libc_csu_fini, ...)`

**4.2 `main()` 函数执行**

`__libc_start_main` 最终调用用户的 `main()` 函数。对于本题 P 程序：

```c
int main() {
    x = 15213;              // ① 将 15213 写入全局变量 x
    y = 0;                  // ② 将 0 写入全局变量 y
    f();                    // ③ 调用 f()，将 x 改为 1.0（double）
    printf("x = %d\n", x);  // ④ 按 %d 格式打印 x
    return 0;               // ⑤ 返回 0
}
```

- ①~③：CPU 执行指令，访问已映射到内存的 `.data`/`.bss` 段，读写全局变量
- ④：`printf` 调用 `write()` 系统调用，将格式化后的字符串 `"x = 某个值\n"` 写入文件描述符 1（标准输出 stdout）。内核将数据通过终端驱动传回给 Shell 所在的终端窗口，最终显示在屏幕上

**注意**：由于 `x` 在 `main11.c` 中是 `int`（4 字节），在 `f11.c` 中是 `double`（8 字节），且 `f()` 执行 `x = 1.0` 按 double 写入，之后 `main` 中按 `%d` 读取，输出结果可能是错误/无意义的值。

### 五、进程终止与回收

**5.1 进程退出**

`main()` 返回后，`__libc_start_main` 调用 `exit(0)`（或 `_exit(0)` 系统调用）：
- 内核释放进程占用的所有资源（内存页、文件描述符、内核数据结构等）
- 进程进入**僵尸态（zombie）**，保留退出状态（exit code）和 PID 在进程表中，等待父进程回收

**5.2 父进程回收**

Shell 此前已调用 `waitpid()`（或 `wait()`）阻塞等待子进程结束。当子进程退出时，内核唤醒 Shell 进程：
- Shell 从 `waitpid()` 返回，读取子进程的退出状态
- 内核清除子进程的 `task_struct`，从进程表中移除，子进程彻底消失
- Shell 显示新的命令提示符，等待下一条命令输入

## （5）一旦可执行文件 P 运行完毕，直接会导致产生什么信号？信号的接收者是谁？对于信号的接收者而言，信号不排队是什么意思？

### 一、产生的信号

可执行文件 P 运行完毕（进程终止）时，内核会向该进程的**父进程**发送 **SIGCHLD** 信号（信号编号 17，在 Linux 中为 `SIGCHLD`）。

SIGCHLD 是子进程状态改变时内核自动发送给父进程的信号，触发条件包括：
- 子进程终止（正常退出或异常终止）
- 子进程被停止（收到 SIGSTOP 等）
- 子进程被继续（收到 SIGCONT）

本题中 P 正常 `return 0` 退出，属于子进程终止，内核发送 SIGCHLD 给父进程。

### 二、信号的接收者

信号的接收者是 P 进程的**父进程**，即**启动 P 的 Shell 进程**（如 bash）。

Shell 在 `fork()` 创建子进程后，通常有两种方式处理子进程终止：

1. **前台作业**：Shell 调用 `waitpid()` 阻塞等待，此时 SIGCHLD 信号的作用较小（Shell 已经在同步等待）。
2. **后台作业**：如果用户执行 `./P &`，Shell 不会阻塞等待，而是依赖 SIGCHLD 信号异步获知子进程已终止，然后调用 `waitpid()` 回收。

本题中 `./P` 是前台执行，Shell 通常阻塞在 `waitpid()` 上，子进程退出时 Shell 从 `waitpid()` 返回，同时也会收到 SIGCHLD 信号。

### 三、"信号不排队"的含义

**信号不排队**是指：Unix/Linux 信号机制中，**同一类型的信号不会被排队**。内核为每个进程维护一个**待处理信号位图（pending signal bitmask）**，每种信号只用一个 bit 表示"是否有该信号待处理"，而不是用一个计数器。

具体含义：

1. **位图而非队列**：当进程收到一个信号时，内核将该信号对应的 bit 置为 1。如果该信号的处理函数尚未执行，又来了一个同类型的信号，内核发现该 bit 已经是 1，**第二个信号就被丢弃了**——不会排队等待。

2. **丢失信号**：例如，如果 Shell 同时有 3 个子进程几乎同时退出，内核会向 Shell 发送 3 次 SIGCHLD。但如果 Shell 还没来得及处理第一个 SIGCHLD，后两个 SIGCHLD 就会因为位图已置位而被**合并/丢弃**。Shell 最终可能只收到 1 次 SIGCHLD 通知，而非 3 次。

3. **对比排队机制**：与之相对的是"信号排队"——如果信号排队，每个信号实例都会被记录在一个队列中，接收者可以逐个处理，不会丢失。POSIX 实时信号（`SIGRTMIN` ~ `SIGRTMAX`）支持排队，但标准信号（如 SIGCHLD）不支持。

4. **实际影响**：Shell 在 SIGCHLD 处理函数中通常采用**循环调用 `waitpid(-1, ..., WNOHANG)`** 的方式来应对信号不排队的问题——一次 SIGCHLD 到达后，Shell 循环回收所有已终止的子进程，直到 `waitpid` 返回 -1（没有更多已终止子进程），从而避免僵尸进程堆积。

**总结**：信号不排队意味着同一标准信号多次产生时，接收者最多只感知到一次，多余的信号实例会被静默丢弃。这是 Unix 信号基于位图实现而非队列实现的直接结果。

| 产生什么信号 | **SIGCHLD**（信号编号 17） |
| 信号接收者 | P 的**父进程**（即 Shell） |
| 信号不排队的含义 | 同一信号类型只用一个 bit 记录，多次发送同一信号时，后续信号被丢弃而不排队，接收者最多感知到一次 |
