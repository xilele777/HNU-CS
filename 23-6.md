# 六. 综合分析题 (25分)

现有如下 C 代码片段：

```c
extern int number = 2;   // 全局变量，初始化为 2（extern + 初始化 = 定义）

int sharedV() { return number; }  // 返回 number 的值（2），用于数组大小和循环次数

int main()
{
    int i, child_status;
    pid_t pid[sharedV];    // pid[2]，存储两个子进程的 PID

    // 信号处理函数（嵌套在 main 内，GCC 扩展语法）
    void myHandler(int sig) {
        printf("Process received signal\n");  // 收到信号时打印
        exit(0);                               // 退出，状态码 0
    }
    signal(SIGINT, myHandler);  // 注册 SIGINT(Ctrl+C) 的处理函数，fork 后子进程继承

    /* ===== 第一个循环：创建子进程 ===== */
    for (i = 0; i < sharedV(); i++) {      // i = 0, 1，共创建 2 个子进程
        if ((pid[i] = fork()) == 0) {      // fork() 返回 0 = 子进程
            while (1);                      // 子进程死循环，等待父进程发信号
        }                                   // 父进程得到子进程 PID，存入 pid[i]
    }
    // 此时：父进程 + 2 个子进程（子进程在 while(1) 中）

    /* ===== 第二个循环：向子进程发送 SIGINT 信号 ===== */
    for (i = 0; i < sharedV(); i++) {
        kill(pid[i], SIGINT);  // 向子进程发送 SIGINT，触发 myHandler → printf + exit(0)
    }

    /* ===== 第三个循环：回收子进程 ===== */
    for (i = 0; i < sharedV(); i++) {
        pid_t wpid = wait(&child_status);  // 阻塞等待任意子进程终止，返回其 PID
        if (WIFEXITED(child_status))       // 检查子进程是否正常退出（exit/return）
            printf("Process %d: Hello,World!%d\n",
                   wpid,                    // 被回收子进程的 PID
                   WEXITSTATUS(child_status)); // 子进程退出状态码（exit(0) → 0）
    }
}
```

请阅读分析上述 C 代码后，尝试解答下列问题：

---

### (1) （5分）这段代码的输出结果可能是什么？为什么会有这样的输出？

---

#### 解答

**可能输出：**

```
Process received signal
Process received signal
Process <pid1>: Hello,World!0
Process <pid2>: Hello,World!0
```

**但存在多种可能情况，按确定性从高到低分析：**

---

##### 确定的部分（一定会发生）

- 父进程最终打印 **2 行** `Process <pid>: Hello,World!0`（`<pid>` 是两个子进程的 PID，0 是 `exit(0)` 的退出码）
- 两个子进程各打印 **1 行** `Process received signal`
- 总共 **4 行输出**

##### 不确定的部分（取决于调度）

**① 子进程两行的先后顺序**

两个子进程是并发执行的，谁先收到信号、谁先执行 `printf`，由内核调度决定，没有固定顺序。

**② 父进程两行的先后顺序**

`wait()` 不指定回收哪个子进程，谁先退出就回收谁。如果子进程 A 先退出，父进程先打印 A 的 PID；反之亦然。

**③ 子进程和父进程的输出是否交错**

两种典型情况：

```
情况一（子进程先全部退出，父进程再打印）：
  Process received signal          ← 子 A
  Process received signal          ← 子 B
  Process 1234: Hello,World!0      ← 父，回收 A
  Process 1235: Hello,World!0      ← 父，回收 B

情况二（一个子进程退出后父进程立即回收，另一子进程还在运行）：
  Process received signal          ← 子 A
  Process 1234: Hello,World!0      ← 父（wait 在 A 退出后立即返回）
  Process received signal          ← 子 B（稍后才收到信号并退出）
  Process 1235: Hello,World!0      ← 父
```

情况二之所以可能发生：父进程 `kill(A)` → `kill(B)` 后立即进入 `wait()`。如果 A 极快处理信号并退出，父进程的第一次 `wait()` 就会立刻返回，此时 B 可能还没来得及处理信号。

---

##### 潜在的异常情况

**④ 输出行数可能少于 4 行**

理论上信号处理函数中调用 `printf` 是**不安全的**（`printf` 不是 async-signal-safe 函数）。如果信号到达时子进程恰好正在执行某个标准库内部操作，可能引发死锁或未定义行为。不过在实际 Linux 环境中，这种简单场景下极少触发。

**⑤ 嵌套函数可能引发的问题**

`myHandler` 是嵌套在 `main()` 中的函数（GCC 扩展），编译器用栈上的 trampoline 实现。如果系统启用了不可执行栈（`NX bit`），理论上可能导致段错误。不过 GCC 会自动为嵌套函数生成 `execstack` 标记，所以通常不会出问题。

---

##### 总结

| 项目 | 是否确定 |
|:---|:---|
| 输出 4 行 | ✅ 确定（正常情况） |
| 每行内容 | ✅ 确定 |
| 子进程两行顺序 | ❌ 依赖调度 |
| 父进程两行顺序 | ❌ 依赖哪个子进程先退出 |
| 父子输出交错 | ❌ 依赖调度 |

### (2) （5分）在这段代码中 `wait` 函数的作用是什么？如果改用 `waitpid` 函数可能会产生什么影响？

---

#### 解答

##### `wait` 的作用

`wait(&child_status)` 做了三件事：

1. **阻塞**：父进程暂停执行，直到**任意一个**子进程终止
2. **回收**：回收已终止子进程的内核资源（PCB、退出状态等），防止僵死进程
3. **获取退出状态**：通过 `child_status` 返回子进程的退出信息，`WIFEXITED` 判断是否正常退出，`WEXITSTATUS` 提取退出码

本题中循环调用两次 `wait()`，每次阻塞等待一个子进程结束，两次正好回收完两个子进程。

##### 改用 `waitpid` 的影响

`waitpid` 比 `wait` 多了两个参数，提供了更精细的控制。不同用法效果不同：

**① `waitpid(pid[i], &child_status, 0)` — 指定 PID**

```c
for (i = 0; i < sharedV(); i++) {
    pid_t wpid = waitpid(pid[i], &child_status, 0);  // 按数组顺序回收
    ...
}
```

**影响**：回收顺序变为**确定性**——必须先回收 `pid[0]`，再回收 `pid[1]`。如果 `pid[0]` 的子进程还没退出，父进程就阻塞等着，哪怕 `pid[1]` 的子进程已经退出了也不会回收。而原来的 `wait()` 是"谁先退出回收谁"。

**② `waitpid(-1, &child_status, 0)` — 等同 `wait`**

与原来行为完全一致，无影响。

**③ `waitpid(-1, &child_status, WNOHANG)` — 非阻塞**

**影响**：如果子进程还没退出，`waitpid` 立即返回 0（而不是阻塞等待）。循环会迅速跑完，可能导致：
- 子进程还没处理完信号、没退出 → `WIFEXITED` 为假 → `printf` 不执行
- 父进程直接结束，子进程变成**孤儿进程**（被 init 收养）
- 子进程的 `exit` 输出和父进程的输出时序完全打乱

**④ 对比总结**

| | `wait()` | `waitpid(pid[i],...,0)` | `waitpid(-1,...,WNOHANG)` |
|:---|:---|:---|:---|
| 回收目标 | 任意子进程 | 指定子进程 | 任意子进程 |
| 阻塞 | 是 | 是 | **否** |
| 回收顺序 | 不确定 | **确定**（按数组顺序） | 不确定 |
| 风险 | 无 | 顺序不对可能死等 | 可能回收不到子进程 |

### (3) （5分）当观察该代码对应的可重定位目标文件时，发现如下信息：

```asm
fc: e8 fc ff ff ff ff      call   fd <main+0xda>
fd: R_386_PC32 kill

108: a1 00 00 00 00       mov    0x0, %eax
109: R_386_32 sharedV
```

请问这两行是什么意思？作用是什么？在可执行文件里面这两行会产生什么变化？

---

#### 解答

这两行是 `.rel.text` 段中的**重定位条目**。本质就是一句话：

> 链接器，在偏移 `X` 处，用 `Y` 方式，填入符号 `Z` 的地址。

---

##### 先看第一条：`R_386_PC32 kill`

题目给了两行，一行是**指令本身**，一行是**修正指令**：

```
fc: e8 fc ff ff ff ff      call   fd <main+0xda>
fd: R_386_PC32 kill
```

**第一步：找到要被修正的位置**

```
fc:  e8  fc ff ff ff ff
 ↑    ↑   ↑
 │    │   └── 操作数（call 后面跟的 4 字节，即 ff ff ff ff = −4，占位用的）
 │    └────── 操作码 e8（1 字节，代表 call 指令）
 └─────────── 这条指令从 0xfc 处开始
```

- `0xfc` = 指令起点
- `0xfc + 1 = 0xfd` = 操作数的起点（跳过 `e8` 这 1 字节）
- 这就是为什么重定位行写的是 `fd:` — 它要修正的是操作数，不是操作码

**第二步：理解修正方式**

```
fd: R_386_PC32 kill
 ↑   ↑          ↑
 │   │          └── 目标符号：填 kill 的地址
 │   └───────────── 重定位类型：PC 相对地址
 └───────────────── 修正位置：0xfd（操作数存放处）
```

- `R_386_PC32`：**PC 相对**。意思是填的不是 kill 的绝对地址，而是 `kill 地址 − 当前指令地址` 的差值（相对偏移）
- 公式：修正值 = **S + A − P**
  - **S** = `kill` 符号在可执行文件中的最终地址（链接后才知道）
  - **A** = 当前占位值 = **−4**（`call` 操作数填 −4 的原因见下文）
  - **P** = 被修正位置 `0xfd` 在可执行文件中的最终地址

> **为什么 A = −4？** `call` 跳转时 CPU 从"下一条指令地址"开始算：`跳转目标 = 下条指令地址 + 操作数`。`call` 全长 5 字节，下条指令地址 = `fc + 5`。如果填 0，`call` 跳到 `fc+5`（可能跑飞）；填 −4，`call` 跳到 `fc+5−4 = fd`（回指自身，无害占位）。所以 −4 是最安全的"地址未知时的默认值"。

> **代入本题实际数字验证：**
>
> 假设链接后代码段加载到 `0x8048000`，则：
>
> ```
> S = kill 的地址        = 0x804a0c0
> A = 占位值             = −4
> P = 0xfd 的最终地址    = 0x8048000 + 0xfd = 0x80480fd
> ```
>
> 链接器计算：
>
> ```
> 修正值 = S + A − P
>        = 0x804a0c0 + (−4) − 0x80480fd
>        = 0x804a0c0 − 0x80480fd − 4
>        = 0x1fc3 − 4
>        = 0x1fbf        ← 写入操作数位置
> ```
>
> CPU 执行时验证：
>
> ```
> 跳转目标 = 下条指令地址 + 操作数
>          = (P + 4) + 修正值
>          = 0x8048101 + 0x1fbf
>          = 0x804a0c0 = kill  ← 正确！
> ```

**第三步：链接前后对比**

```
链接前 (.o):  e8 fc ff ff ff    →  call 0xfd（跳转到自己，显然不对，占位而已）
链接后 (exe): e8 XX XX XX XX    →  call 804a0c0 <kill>（填了 kill 的真实相对偏移）
```

---

##### 再看第二条：`R_386_32 sharedV`

```
108: a1 00 00 00 00       mov    0x0, %eax
109: R_386_32 sharedV
```

**同样三步拆解：**

```
108:  a1  00 00 00 00
 ↑     ↑   ↑
 │     │   └── 操作数 = 00 00 00 00（占位值 0，因为 sharedV 地址未知）
 │     └────── a1 = mov 绝对地址, %eax（从内存绝对地址处读 4 字节到 eax）
 └──────────── 指令从 0x108 开始
```

- 操作码 `a1` 占 1 字节，所以操作数从 `0x108 + 1 = 0x109` 开始
- 重定位行 `109: R_386_32 sharedV` 就是要修正 0x109 处的 4 字节

```
109: R_386_32 sharedV
 ↑    ↑         ↑
 │    │         └── 目标符号：填 sharedV 的绝对地址
 │    └──────────── 重定位类型：绝对地址
 └───────────────── 修正位置：0x109
```

- `R_386_32`：**绝对地址**。直接填 sharedV 的最终地址，不做相对计算
- 公式：修正值 = **S + A**（S = sharedV 地址，A = 当前占位值 0）

**链接前后对比：**

```
链接前 (.o):  a1 00 00 00 00    →  mov 0x0, %eax（从地址 0 读，显然不对）
链接后 (exe): a1 XX XX XX XX    →  mov 0x804a018, %eax（填了 sharedV 的真实地址）
```

---

##### 一张表总结

| | 第一条 | 第二条 |
|:---|:---|:---|
| 指令 | `call` | `mov` |
| 操作码 | `e8`（1 字节） | `a1`（1 字节） |
| 操作数位置 | `0xfd` | `0x109` |
| 占位值 | `ff ff ff ff`（−4） | `00 00 00 00`（0） |
| 重定位类型 | `R_386_PC32` | `R_386_32` |
| 填什么 | kill 的**相对偏移** | sharedV 的**绝对地址** |

> **核心规律**：操作码 1 字节 → 操作数偏移 = 指令偏移 + 1。重定位行就是告诉链接器：到这个操作数偏移处，按指定的类型，把符号地址填进去。

---

##### 扩展：为什么 `call` 用相对地址，`mov` 用绝对地址？

**用指路来类比：**

- **绝对地址** = "去北京市海淀区中关村大街 59 号" → 换一个城市就废了
- **相对地址** = "往前走 200 米左转" → 不管你在哪，走 200 米就到

**为什么 `call` 用相对地址：**

假设代码被加载到内存的不同位置：

```
情况一：代码加载到 0x1000
  call 指令在 0x1000，kill 在 0x2000
  距离 = 0x2000 − 0x1000 = 0x1000

情况二：代码加载到 0x5000（共享库场景）
  call 指令在 0x5000，kill 在 0x6000
  距离 = 0x6000 − 0x5000 = 0x1000  ← 距离不变！
```

`call` 和 `kill` 之间的**距离是固定的**，不管代码搬到哪里。所以存距离比存绝对地址灵活——代码搬到哪都能正确跳转。这就是**位置无关代码（PIC）** 的基础。

**CPU 执行 `call` 的完整过程：**

```
地址:    fc    fd    fe    ff   100   101
        ┌─────────────────────┐
        │  call 指令 (5字节)   │  ← 下条指令从这里开始
        └─────────────────────┘
         ↑                    ↑
       当前 PC            PC+5（下条指令地址）
```

跳转目标 = **下条指令地址 + 操作数**

```
链接前：操作数 = −4 → 跳转目标 = 101 + (−4) = fd（回指自身，无害）
链接后：操作数 = kill − 101 → 跳转目标 = 101 + (kill−101) = kill（正确！）
```

**为什么 `mov` 用绝对地址：**

`mov absolute_address, %eax` 加载的是全局变量/函数指针，这些符号在内存中有**固定的绝对地址**（由链接器最终确定），不存在"搬来搬去"的问题。所以直接填绝对地址即可。

> **一句话**：相对重定位存"距离"，代码搬哪都能用；绝对重定位存"地址"，放那就不能动了。

### (4) （5分）对可执行文件进行反汇编观察时，发现在 `<main>` 部分对应 `printf` 函数调用的地方出现了如下汇编代码：

```asm
call 80483e0 <printf@plt>
```

继而查看 `0x80483e0` 地址处的内容，发现出现如下语句：

```asm
jmp *0x804a00c
push $0x0
jmp 80483d0 <_init+0x24>
```

请结合你学习的 PIC 相关知识尝试解答这几条语句是干什么用的，以及后续对 `printf` 的重定位过程。

---

#### 解答

##### 问题：为什么不能直接 call printf？

`printf` 的代码在 `libc.so` 动态库里。这个库每次加载到内存的地址都不一样，而且被所有进程共享。编译时谁也不知道 printf 的实际地址，也没法在代码里硬编码——代码段是只读共享的，改了别的进程就崩了。

**于是发明了两样东西：PLT + GOT。**

---

##### PLT 和 GOT 各是什么

```
┌─── 代码段（只读，所有进程共享）───┐   ┌─── 数据段（可写，每个进程独有）───┐
│                                  │   │                               │
│  PLT（跳板）                      │   │  GOT（地址表）                 │
│  ┌────────────────────────┐      │   │  ┌──────────────────────┐     │
│  │ printf@plt:            │      │   │  │ 0x804a00c: [  ?  ]   │     │
│  │   jmp *0x804a00c ─────────────│──│──→  存 printf 的真实地址  │     │
│  │   push $0x0            │      │   │  └──────────────────────┘     │
│  │   jmp 解析器            │      │   │                               │
│  └────────────────────────┘      │   │                               │
│                                  │   │                               │
│  call printf@plt ──────────────→ │   │                               │
│  （代码里只写 PLT 的地址，        │   │                               │
│    永远不变，不用改代码段）        │   │                               │
└──────────────────────────────────┘   └───────────────────────────────┘
```

- **PLT**：代码段里的**跳板**。每个外部函数一个 PLT 入口，内容固定不变
- **GOT**：数据段里的**地址表**。每个外部函数一个 GOT 槽位，存真实地址。因为是数据段，**随时可以改**

---

##### 类比：酒店前台（PLT）+ 通讯录（GOT）

你要找一个人，但你不知道他住哪个房间（printf 的真实地址），而且你不能去每个楼层挨个找。

- **GOT** = 通讯录，一页写一个人的房间号。新来的人房间号是空的
- **PLT** = 前台服务员，"我帮你查通讯录"
- **动态链接器** = 大堂经理，知道所有人的房间号

---

##### 第一次找 printf（GOT 里还是空的）

```
call printf@plt                你跟前台说："我找 printf"

① jmp *0x804a00c              前台翻开通讯录 GOT 中 printf 的槽位
      GOT 中 printf 的槽位 = 下一条指令       通讯录写着"还没登记，去问经理"（指回 PLT 自己）

② push $0x0                   前台把"printf"的编号递给经理

③ jmp 解析器                   前台喊经理来解决
      │
      经理查出 printf 的房间号
      把房间号填进通讯录 GOT 中 printf 的槽位
      带你去找 printf
```

---

##### 第二次找 printf（GOT 里已经填好了）

```
call printf@plt                你又跟前台说："找 printf"

① jmp *0x804a00c              前台翻开通讯录 GOT 中 printf 的槽位
      GOT 中 printf 的槽位 = printf 真实地址  通讯录已经有了！
      直接带你去 printf         无需惊动经理
```

---

##### 对应回题目指令

```asm
call 80483e0 <printf@plt>      ← 代码里只写 PLT 的地址，永远不动

printf@plt:
    jmp *0x804a00c             ← 查 GOT 中 printf 的槽位：有就直达，没有就穿透
    push $0x0                  ← 报编号找经理
    jmp 80483d0                ← 找经理（动态链接器）去解析
```

---

##### 一句话

> **PLT 是固定跳板（代码不改），GOT 是可变地址表（数据可改）。** 第一次走经理查房号填表，第二次直接走表。整个过程只改数据段（GOT），代码段纹丝不动——这就是 PIC 的精髓。

### (5) （5分）对于题干中的代码，不考虑数据结构和算法的变更，做什么修订可能能改善该代码的执行性能？

---

#### 解答

##### ① 最大的性能问题：`while(1)` 忙等待

子进程在收到信号前一直执行 `while(1)`，**100% 占用 CPU**，白白浪费处理器时间。应改为 `pause()`：

```c
// 原代码（差）：
if ((pid[i] = fork()) == 0) {
    while (1);     // 空转烧 CPU
}

// 改进（好）：
if ((pid[i] = fork()) == 0) {
    pause();       // 挂起进程，0% CPU，收到信号才唤醒
}
```

`pause()` 使进程进入**阻塞态**，不消耗 CPU 时间片。收到 SIGINT 后内核唤醒进程，信号处理函数执行，`exit(0)` 退出。这也是"异常控制流"章节的核心思想：**用阻塞替代忙等**。

##### ② 循环中重复调用 `sharedV()`

三个 for 循环每次都调用 `sharedV()` 来判断循环条件：

```c
for (i = 0; i < sharedV(); i++)   // 每次迭代都调用 sharedV()
```

虽然 `sharedV()` 只是 `return number`，但函数调用仍有开销（压栈、跳转、返回）。改为局部变量缓存：

```c
int n = sharedV();                // 只调一次，存起来
for (i = 0; i < n; i++) { ... }
for (i = 0; i < n; i++) { ... }
for (i = 0; i < n; i++) { ... }
```

##### ③ 信号处理函数中的 `printf` 不安全

`printf` 不是异步信号安全函数。在信号处理函数中调用 `printf` 可能导致死锁或输出混乱。改用 `write()`：

```c
// 原代码（不安全）：
void myHandler(int sig) {
    printf("Process received signal\n");  // 非异步信号安全
    exit(0);
}

// 改进（安全）：
void myHandler(int sig) {
    write(1, "Process received signal\n", 24);  // 异步信号安全
    _exit(0);  // _exit 也是异步信号安全的
}
```

---

##### 总结

| 问题 | 修订 | 原理 |
|:---|:---|:---|
| `while(1)` 忙等 | 改为 `pause()` | 阻塞替代忙等，释放 CPU |
| 循环中重复调用 `sharedV()` | 缓存到局部变量 | 减少函数调用开销 |
| 信号处理函数用 `printf` | 改为 `write()` + `_exit()` | 使用异步信号安全函数 |
