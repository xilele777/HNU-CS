## 📋 题目代码

### `compute.c`

```c
extern int global_factor; // 在其他文件中定义的全局变量

// 计算数组元素的加权平方和
int compute_sum_squares(int *array, int size) {
    int sum = 0;
    for (int i = 0; i < size; i++) {
        sum += array[i] * array[i] * global_factor;
    }
    return sum;
}
```

### `globals.c`

```c
int global_factor = 2; // 全局变量定义
```

### `child_handler.c`

```c
#include <stdio.h>
#include <sys/wait.h>
#include <unistd.h>

void handle_child_exit() {
    int status;
    pid_t pid;

    pid = wait(&status);
    if (pid > 0) {
        printf("Child %d exited with status %d\n", pid,
               WEXITSTATUS(status));
    }
}
```

### `main.c`

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <signal.h>

#define ARRAY_SIZE 1000
#define NUM_CHILDREN 3

// 函数与变量声明
extern int compute_sum_squares(int *array, int size);
extern int global_factor;
extern void handle_child_exit();

void sigchld_handler(int sig) {
    printf("Received SIGCHLD signal\n");
    handle_child_exit();
}

int main() {
    signal(SIGCHLD, sigchld_handler);
    int *data = malloc(ARRAY_SIZE * sizeof(int));
    for (int i = 0; i < ARRAY_SIZE; i++) {
        data[i] = i % 100;
    }
    global_factor = 3;
    for (int i = 0; i < NUM_CHILDREN; i++) {
        pid_t pid = fork();
        if (pid == 0) {
            int result = compute_sum_squares(data, ARRAY_SIZE);
            printf("Child %d: Sum of squares: %d\n", getpid(), result);
            exit(i);
        } else {
            printf("Created child process with PID: %d\n", pid);
            usleep(10000); // 挂起 10000 微秒，确保子进程启动
        }
    }
    sleep(3);
    free(data);
    printf("Parent process exiting\n");
    return 0;
}
```

### 1) `main.c` 如何生成 `main.o`？`.o` 文件的基本格式是什么？包含哪些符号？强弱属性如何？（6 分）

#### 🔧 生成过程

`main.c` → `main.o` 经过 **预处理 → 编译 → 汇编** 三个阶段：

```bash
gcc -c main.c -o main.o
```

| 阶段 | 命令 | 输入 | 输出 | 工作内容 |
|------|------|------|------|----------|
| 预处理 | `gcc -E` | `main.c` | `main.i` | 展开 `#include` 头文件、替换 `#define` 宏、删除注释 |
| 编译 | `gcc -S` | `main.i` | `main.s` | 将 C 代码翻译为 x86-64 汇编代码 |
| 汇编 | `gcc -c` | `main.s` | `main.o` | 将汇编指令翻译为机器码，生成 **可重定位目标文件（ELF Relocatable Object File）** |

#### 📦 `.o` 文件的基本格式（ELF 结构）

`main.o` 是 **ELF 可重定位目标文件**，大致分为以下几部分：

```text
┌──────────────────────────────┐
│        ELF 头 (ELF Header)    │  ← 魔数、文件类型、机器类型、入口等元信息
├──────────────────────────────┤
│       .text   (代码段)        │  ← 编译后的机器指令（main、sigchld_handler）
├──────────────────────────────┤
│       .data   (数据段)        │  ← 已初始化的全局/静态变量
├──────────────────────────────┤
│       .bss    (BSS段)         │  ← 未初始化的全局/静态变量（占位，不占磁盘空间）
├──────────────────────────────┤
│       .rodata (只读数据)       │  ← 字符串常量（如 "Received SIGCHLD..." 等）
├──────────────────────────────┤
│       .symtab (符号表)         │  ← 记录模块中定义和引用的符号
├──────────────────────────────┤
│       .rel.text (代码重定位表)  │  ← .text 中需要重定位的位置（函数调用等）
├──────────────────────────────┤
│       .rel.data (数据重定位表)  │  ← .data 中需要重定位的位置（全局变量引用等）
├──────────────────────────────┤
│       .comment / .debug 等    │  ← 调试信息、注释段等
├──────────────────────────────┤
│     节头表 (Section Header)    │  ← 描述每个节的偏移、大小、属性
└──────────────────────────────┘
```

`main.o` 的符号表中包含以下符号：

| 符号名 | 类型 | 强弱属性 | 所在模块 | 说明 |
|--------|------|----------|----------|------|

| `handle_child_exit` | 外部引用 (UNDEF, GLOBAL) | 未定义（引用） | 需从 child_handler.c 解析 | `extern` 声明，链接时解析 |

### 2) 请分别描述对 `global_factor` 和 `compute_sum_squares` 这两个符号引用，如何进行重定位？（4 分）

重定位是链接器的核心工作之一：在各 `.o` 文件的节合并并分配最终地址后，修改代码/数据中的地址字段，使引用指向正确的位置。

#### 📍 `global_factor` —— 数据符号的 PC 相对重定位

- **符号来源**：在 `globals.c` 中定义 (`int global_factor = 2;`)，属于 `.data` 节
- **引用位置**：出现在 `main.c` 的 `global_factor = 3;` 语句中，编译为一条 **PC 相对寻址的 mov 指令**（x86-64 下数据访问通常使用 `%rip` 相对寻址）
- **重定位类型**：`R_X86_64_PC32`（PC 相对重定位，32 位偏移）

**重定位过程：**

```text
原始汇编（main.o）：  movl $3, global_factor(%rip)
                            ↑ 此处偏移暂为 0 或占位值

链接时计算填入值 = S + A - P

其中：
  S = global_factor 在可执行文件中的最终虚拟地址（链接器分配）
  A = 加数（addend），即指令中已有常数偏移（通常为 -4，因为 %rip 指向下一条指令）
  P = 被重定位位置（该 mov 指令中偏移字段所在地址）

最终效果：指令中的偏移被修正，运行时 %rip + offset = global_factor 的地址
```

#### 📍 `compute_sum_squares` —— 函数符号的 PC 相对重定位

- **符号来源**：在 `compute.c` 中定义，属于 `.text` 节
- **引用位置**：出现在 `main.c` 的 `compute_sum_squares(data, ARRAY_SIZE);` 调用中，编译为 `call` 指令
- **重定位类型**：`R_X86_64_PC32` 或 `R_X86_64_PLT32`（通过 PLT 时）

**重定位过程：**

```text
原始汇编（main.o）：  call compute_sum_squares
                            ↑ 此处偏移暂为 0 或占位值

链接时计算填入值 = S + A - P

其中：
  S = compute_sum_squares 函数入口的最终虚拟地址（.text 节中）
  A = 加数（addend）
  P = call 指令中偏移字段所在地址

最终效果：call 指令中的相对偏移被修正，运行时 call 指令能正确跳转到 compute_sum_squares 的入口
```

#### 📊 两种重定位对比

| 对比维度 | `global_factor` | `compute_sum_squares` |
|----------|:--------------:|:---------------------:|
| 符号类型 | 数据符号 | 函数符号 |
| 定义所在节 | `.data` | `.text` |
| 引用指令 | `movl $3, X(%rip)` | `call X` |
| 重定位类型 | `R_X86_64_PC32` | `R_X86_64_PC32` / `R_X86_64_PLT32` |
| 重定位公式 | `S + A - P` | `S + A - P` |
| 本质 | 修正数据地址偏移 | 修正跳转目标偏移 |

> **核心公式复习**：
> - **绝对重定位**：填入 `S + A`
> - **PC 相对重定位**：填入 `S + A - P`
> - 其中 S = 符号最终地址，A = 加数，P = 被修改位置地址

### 4) `main.c` 中对于 `SIGCHLD` 信号处理会出现什么问题？为什么？怎么解决？（5 分）

#### 🔴 存在的问题

**问题一：信号不排队导致僵死进程（核心问题）**

SIGCHLD 属于传统 Unix 信号，**同一信号在 pending 期间只保留一个标记**。当多个子进程几乎同时退出时：

```text
子进程 A 退出 → SIGCHLD 发送 → pending = 1
子进程 B 退出 → SIGCHLD 发送 → 发现已 pending，合并（丢失）
子进程 C 退出 → SIGCHLD 发送 → 发现已 pending，合并（丢失）

信号处理函数被调用 1 次 → handle_child_exit() → wait() 只回收 1 个子进程
→ 另外 2 个子进程退出状态未被回收 → 成为僵死进程（zombie）
```

**问题二：信号处理函数中使用非异步信号安全函数**

`sigchld_handler` 和 `handle_child_exit` 中调用了 `printf()`。`printf` **不是异步信号安全的**（async-signal-safe），在信号处理上下文中调用可能导致：

- 与主程序的 `printf` 发生死锁（争抢内部锁）
- 输出内容交错混乱
- 未定义行为

**问题三：`wait()` 可能阻塞信号处理函数**

`handle_child_exit()` 中的 `wait(&status)` 是**阻塞式**调用。如果某个子进程尚未退出但 SIGCHLD 已被触发（例如信号来自其他原因），`wait()` 会让信号处理函数一直阻塞，导致整个进程卡住。

#### 🟢 解决方案

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <signal.h>
#include <sys/wait.h>
#include <errno.h>

void sigchld_handler(int sig) {
    int old_errno = errno;    // ① 保存 errno，避免破坏主程序
    int status;
    pid_t pid;

    // ② 使用 waitpid + WNOHANG 循环回收所有已退出的子进程
    while ((pid = waitpid(-1, &status, WNOHANG)) > 0) {
        // ③ 不使用 printf（非异步安全），改用 write 或仅设置标志
    }

    errno = old_errno;        // ④ 恢复 errno
}
```

**修复要点总结：**

| 问题 | 原因 | 解决方法 |
|------|------|----------|
| 僵死进程 | 信号不排队，多个子进程退出只触发一次处理 | 用 `while(waitpid(-1, &status, WNOHANG) > 0)` **循环非阻塞回收** |
| 非异步安全 | `printf` 不是 async-signal-safe | 避免在 handler 中使用 `printf`，可改用 `write()` 系统调用或设置 `volatile sig_atomic_t` 标志位 |
| 阻塞风险 | `wait()` 是阻塞调用 | 使用 `waitpid()` + `WNOHANG` 标志实现**非阻塞**回收 |
| errno 被覆盖 | handler 可能改变 errno | 在 handler 开头保存 `old_errno`，末尾恢复 |

> **关键知识串联**：本题综合考查了信号机制（第 44 问）、进程管理（第 42-43 问）、异步信号安全等知识点。

### 5) 对于 `compute.c` 中的代码，如何利用课程优化知识提高指令级并行性能？（5 分）

#### 🔍 原始代码的性能瓶颈

```c
int compute_sum_squares(int *array, int size) {
    int sum = 0;
    for (int i = 0; i < size; i++) {
        sum += array[i] * array[i] * global_factor;
    }
    return sum;
}
```

**核心问题：循环携带依赖（Loop-Carried Dependency）**

```text
迭代 0: sum = sum₀ + array[0]² × global_factor
         ↓ 必须等 sum 算完
迭代 1: sum = sum₁ + array[1]² × global_factor
         ↓ 必须等 sum 算完
迭代 2: ...
```

每次迭代的 `sum += ...` 都依赖上一次迭代的 `sum` 值，形成一条**串行关键路径**。CPU 流水线被迫停顿等待前一次加法完成，无法充分利用指令级并行（ILP）。

**次要问题：全局变量阻碍优化**

`global_factor` 是全局变量（`extern`），编译器必须假设：
- 每次循环迭代中 `global_factor` 可能被其他代码修改（例如信号处理函数）
- 不能将 `global_factor` 缓存到寄存器中
- 不能将 `global_factor * array[i] * array[i]` 的计算重排

这导致每次迭代都需要**从内存重新读取 `global_factor`**。

#### ✅ 优化方案：多路并行累加 + 循环展开 + 全局变量提升

```c
int compute_sum_squares(int *array, int size) {
    int sum0 = 0, sum1 = 0, sum2 = 0, sum3 = 0;  // ① 多个独立累加器
    int i;
    int factor = global_factor;                     // ② 将全局变量读入局部变量
    int limit = size - 3;                           // ③ 计算展开边界

    // ④ 4 路并行累加，每次处理 4 个元素
    for (i = 0; i < limit; i += 4) {
        sum0 += array[i]     * array[i]     * factor;
        sum1 += array[i + 1] * array[i + 1] * factor;
        sum2 += array[i + 2] * array[i + 2] * factor;
        sum3 += array[i + 3] * array[i + 3] * factor;
    }

    // ⑤ 处理尾部剩余元素
    for (; i < size; i++) {
        sum0 += array[i] * array[i] * factor;
    }

    // ⑥ 合并结果
    return sum0 + sum1 + sum2 + sum3;
}
```

#### 📊 优化原理分析

| 优化技术 | 解决的问题 | 原理 |
|----------|-----------|------|
| **多路并行累加** | 串行关键路径依赖 | 4 个独立累加器形成 4 条**相互独立**的依赖链，CPU 可同时发射多条浮点/整数乘法与加法指令 |
| **循环展开 (4×)** | 循环开销 + 依赖密度 | 减少循环控制指令（`cmp`、`jmp`）比例；展开后可更方便地交错安排独立指令 |
| **全局变量提升** | 每次迭代读内存 | `int factor = global_factor;` 将全局变量值缓存到寄存器中，消除循环内重复 load |

**关键路径对比：**

```text
优化前（单路）：
  sum → sum → sum → ... → sum   （一条长链，N 个节点串行）
  关键路径长度 = N × (乘法延迟 + 加法延迟)

优化后（四路）：
  sum0 → sum0 → ... → sum0  }
  sum1 → sum1 → ... → sum1  }  四条短链并行
  sum2 → sum2 → ... → sum2  }  关键路径长度 ≈ N/4 × (乘法延迟 + 加法延迟)
  sum3 → sum3 → ... → sum3  }

  最后：sum = sum0 + sum1 + sum2 + sum3 （额外 2-3 次加法）
```

**为什么单纯循环展开不够？**

如果只展开循环但只用一个累加器：

```c
sum += a[i]*a[i]*factor + a[i+1]*a[i+1]*factor + ...
```

这等价于依然是单条依赖链，瓶颈在于**延迟（latency）**而非吞吐量（throughput）。而多路并行利用了现代 CPU 可以**每个时钟周期启动一条新指令**的能力，让吞吐量接近上限。

> **课程知识对应**：本题涉及"程序性能优化"章节（第 46 问：关键路径与多路并行；第 48 问：全局变量阻碍优化）。
