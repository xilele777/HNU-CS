## 五. 综合应用题 (20分)

给定一个 32 位 Linux 系统，其高速缓存（Cache）的大小为 **64 字节**，采用 **2 路组相联**，每个块（Cache Line）大小为 **16 字节**。高速缓存采用的写策略是**直写（Write-through）**，替换策略为 **LRU（最近最少使用）**。

---

### A) 基础知识 (2分)

- **A1)** 高速缓存中有多少个组？

- **A2)** 每个组中有多少个缓存行？

---

#### 解答

**A1) 高速缓存中有多少个组？**

**答案：2 个组。**

计算过程：

- 缓存行（Cache Line）总数 = Cache 总大小 ÷ 块大小 = $64 \div 16 = 4$ 个
- 该 Cache 为 **2 路组相联**，即每个组包含 2 个缓存行
- 组数 = 缓存行总数 ÷ 每组路数 = $4 \div 2 = 2$ 个组

> 公式：$\text{组数} = \dfrac{\text{Cache 总大小}}{\text{块大小} \times \text{相联度}} = \dfrac{64}{16 \times 2} = 2$

---

**A2) 每个组中有多少个缓存行？**

**答案：2 个缓存行。**

"2 路组相联"（2-way set associative）中，"路数"（way）的定义就是每组中缓存行的数量。因此每组有 **2 个**缓存行——这是由相联度直接决定的，无需额外计算。

> 补充：对于这个 Cache，地址划分如下（假设按字节编址）：
> - **块内偏移（Block Offset）**：块大小 $16 = 2^4$ 字节 → **4 位**
> - **组索引（Set Index）**：共 2 个组 → **1 位**
> - **标记（Tag）**：剩余地址位，用于判断是否命中

---

### B) 缓存命中分析 (10分)

**假设如下：**

- 访问一次内存消耗 $100\text{ ns}$
- 访问一次 Cache 的开销是 $1\text{ ns}$
- 忽略可能发生的其他时间开销（回收、存储到高速缓存等）
- 如果使用高速缓存，我们总是会先访问高速缓存（因此，在这种情况下，高速缓存未命中的开销为 $101\text{ ns}$）

给定一个整数数组（假设 `int` 占 4 字节）：

```c
int Arr[6][4];
```

且 `Arr` 数组从地址 `0x00000000` 开始存放。

如果我们要逐一访问数组中的以下元素，请回答：

**B1)** 请用 **"H"** 或 **"M"** 填空，分别表示高速缓存命中（Hit）和高速缓存未命中（Miss）：

| 访问元素 (Access) | 缓存结果 (Cache Result) |
| :---: | :---: |
| `Arr[0][0]` | |
| `Arr[0][2]` | |
| `Arr[0][3]` | |
| `Arr[1][1]` | |
| `Arr[1][3]` | |
| `Arr[2][1]` | |

- **B2)** 如果不使用高速缓存，时间开销是多少？

- **B3)** 如果使用高速缓存，时间开销是多少？

---

#### 解答

##### 前置分析：地址映射

首先明确 Cache 的地址划分：

- **块内偏移**：块大小 $16 = 2^4$ 字节 → **4 位**（地址位 `[3:0]`）
- **组索引**：共 2 个组 → **1 位**（地址位 `[4]`）
- **标记（Tag）**：剩余高位（地址位 `[31:5]`）

每个块 16 字节，可容纳 $16 \div 4 = 4$ 个 `int`。

`Arr[6][4]` 按行优先存储，`Arr[i][j]` 的地址为：

$$\text{Addr} = \underbrace{\text{0x00000000}}_{\text{基地址}} + \underbrace{(i \times 4 + j)}_{\text{第几个元素}} \times \underbrace{4}_{\text{sizeof(int)}}$$

各元素地址、所属块、组、Tag 如下：

| 元素 | 地址 | 所属块 (16B) | 组索引 (bit 4) | Tag |
|:---|:---|:---|:---:|:---|
| `Arr[0][0]` | `0x00` | Block 0 (`0x00–0x0F`) | 0 | 0 |
| `Arr[0][1]` | `0x04` | Block 0 | 0 | 0 |
| `Arr[0][2]` | `0x08` | Block 0 | 0 | 0 |
| `Arr[0][3]` | `0x0C` | Block 0 | 0 | 0 |
| `Arr[1][0]` | `0x10` | Block 1 (`0x10–0x1F`) | 1 | 0 |
| `Arr[1][1]` | `0x14` | Block 1 | 1 | 0 |
| `Arr[1][2]` | `0x18` | Block 1 | 1 | 0 |
| `Arr[1][3]` | `0x1C` | Block 1 | 1 | 0 |
| `Arr[2][0]` | `0x20` | Block 2 (`0x20–0x2F`) | 0 | 1 |
| `Arr[2][1]` | `0x24` | Block 2 | 0 | 1 |
| `Arr[2][2]` | `0x28` | Block 2 | 0 | 1 |
| `Arr[2][3]` | `0x2C` | Block 2 | 0 | 1 |

---

##### 附：地址切分详解（手算方法）

32 位地址从右往左切成三段：

```
┌──────────────────────┬─────┬──────────┐
│        Tag           │ Set │  Offset  │
│     bits [31:5]      │[4]  │  [3:0]   │
│       27位           │ 1位 │   4位    │
└──────────────────────┴─────┴──────────┘
```

- **Offset**：块 16B，定位块内第几个字节，$\log_2 16 = 4$ 位 → 最低 4 位
- **Set**：共 2 组，$\log_2 2 = 1$ 位 → bit 4
- **Tag**：剩下的高位全归 Tag，用来区分同一 Set 里的不同块

**示例 ① `Arr[0][0]` — 地址 `0x00`**

```
0x00 = 0000 0000 0000 0000 0000 0000 0000 0000

  Offset[3:0] = 0000 = 0  → 块内第 0 字节
  Set[4]      =    0 = 0  → 去 Set 0
  Tag[31:5]   = 全0  = 0  → Tag = 0
```

**示例 ② `Arr[1][0]` — 地址 `0x10`**

```
0x10 = 0000 0000 0000 0000 0000 0000 0001 0000

  Offset[3:0] = 0000 = 0  → 块内第 0 字节
  Set[4]      =    1 = 1  → 去 Set 1（bit4 变了！）
  Tag[31:5]   = 全0  = 0  → Tag = 0
```

**示例 ③ `Arr[2][0]` — 地址 `0x20`**

```
0x20 = 0000 0000 0000 0000 0000 0000 0010 0000

  Offset[3:0] = 0000 = 0  → 块内第 0 字节
  Set[4]      =    0 = 0  → 去 Set 0（bit4 又归 0）
  Tag[31:5]   = ...001 = 1 → Tag = 1（bit5 变了！）
```

**规律**：每隔 16 字节（一个块）bit4 翻转一次 → Set 在 0/1 间交替；每隔 32 字节（一组容量）高位进位 → Tag 变化。

---

##### B1) 逐一模拟缓存访问

**初始状态**：Set 0 和 Set 1 均为空（LRU 用最近最少使用顺序表示）。

---

**① 访问 `Arr[0][0]`**（地址 `0x00`，Set 0，Tag 0）

Set 0 中无 Tag 0 → **Miss（M）**

加载 Block 0 到 Set 0 的空闲路（Way 0）：

| Set 0 | Way 0: Tag=0 | Way 1: 空 |

---

**② 访问 `Arr[0][2]`**（地址 `0x08`，Set 0，Tag 0）

Set 0 的 Way 0 命中 Tag 0 → **Hit（H）**

---

**③ 访问 `Arr[0][3]`**（地址 `0x0C`，Set 0，Tag 0）

Set 0 的 Way 0 命中 Tag 0 → **Hit（H）**

---

**④ 访问 `Arr[1][1]`**（地址 `0x14`，Set 1，Tag 0）

Set 1 为空 → **Miss（M）**

加载 Block 1 到 Set 1 的空闲路（Way 0）：

| Set 1 | Way 0: Tag=0 | Way 1: 空 |

---

**⑤ 访问 `Arr[1][3]`**（地址 `0x1C`，Set 1，Tag 0）

Set 1 的 Way 0 命中 Tag 0 → **Hit（H）**

---

**⑥ 访问 `Arr[2][1]`**（地址 `0x24`，Set 0，Tag 1）

Set 0 中：Way 0 有 Tag=0，Way 1 为空。Tag 1 不在 Set 0 → **Miss（M）**

加载 Block 2 到 Set 0 的空闲路（Way 1）：

| Set 0 | Way 0: Tag=0 | Way 1: Tag=1 |

---

**B1 答案汇总：**

| 访问元素 | 缓存结果 |
|:---:|:---:|
| `Arr[0][0]` | **M** |
| `Arr[0][2]` | **H** |
| `Arr[0][3]` | **H** |
| `Arr[1][1]` | **M** |
| `Arr[1][3]` | **H** |
| `Arr[2][1]` | **M** |

---

##### B2) 不使用高速缓存

每次访问都直接访问内存，每次 $100\text{ ns}$。

$$\text{总时间} = 6 \times 100 = \mathbf{600\text{ ns}}$$

---

##### B3) 使用高速缓存

- **Miss**：先访问 Cache（1 ns）再访问内存（100 ns）= **101 ns/次**
- **Hit**：仅访问 Cache = **1 ns/次**

共 3 次 Miss + 3 次 Hit：

$$\text{总时间} = 3 \times 101 + 3 \times 1 = 303 + 3 = \mathbf{306\text{ ns}}$$

### C) 程序循环分析 (8分)

如果我们使用以下程序访问该数组：

```c
int i, j;
for (i = 0; i < 4; i++) {
    for (j = 0; j < 6; j++) {
        Arr[j][i]++;
    }
}
```

请回答：

- **C1)** 如果不使用高速缓存，时间开销是多少？

- **C2)** 如果使用高速缓存，时间开销是多少？

---

#### 解答

##### 访问模式分析

外层循环 `i` 从 0 到 3，内层循环 `j` 从 0 到 5，访问 `Arr[j][i]`：

```
i=0: Arr[0][0] → Arr[1][0] → Arr[2][0] → Arr[3][0] → Arr[4][0] → Arr[5][0]
i=1: Arr[0][1] → Arr[1][1] → Arr[2][1] → Arr[3][1] → Arr[4][1] → Arr[5][1]
i=2: Arr[0][2] → Arr[1][2] → Arr[2][2] → Arr[3][2] → Arr[4][2] → Arr[5][2]
i=3: Arr[0][3] → Arr[1][3] → Arr[2][3] → Arr[3][3] → Arr[4][3] → Arr[5][3]
```

共 $4 \times 6 = 24$ 次访问。这是**按列遍历**——每次访问跳一整行（16 字节），进入不同的块。

##### 块与 Set 映射（扩展至全部 6 行）

| 块 | 地址范围 | 包含元素 | Set | Tag |
|:---|:---|:---|:---:|:---:|
| Block 0 | `0x00–0x0F` | `Arr[0][0]` ~ `Arr[0][3]` | 0 | 0 |
| Block 1 | `0x10–0x1F` | `Arr[1][0]` ~ `Arr[1][3]` | 1 | 0 |
| Block 2 | `0x20–0x2F` | `Arr[2][0]` ~ `Arr[2][3]` | 0 | 1 |
| Block 3 | `0x30–0x3F` | `Arr[3][0]` ~ `Arr[3][3]` | 1 | 1 |
| Block 4 | `0x40–0x4F` | `Arr[4][0]` ~ `Arr[4][3]` | 0 | 2 |
| Block 5 | `0x50–0x5F` | `Arr[5][0]` ~ `Arr[5][3]` | 1 | 2 |

##### 关键冲突分析

Set 0 映射了 3 个块（Block 0、2、4），但只有 2 路。Set 1 同理（Block 1、3、5）。

按列访问时，每次访问都跳到不同块：

```
i=0 时，依次访问 Block 0, 1, 2, 3, 4, 5（全部 Miss，首次加载）
```

到 i=1 时，需要再次访问 Block 0——但 Block 0 早已被 Block 4 踢出 Set 0。**每一轮外层循环，之前的块都被踢光了**，导致全部 Miss。

##### 逐轮 LRU 追踪（简化版）

以 Set 0 为例：

| 时刻 | 访问 | Set 0 状态 | 结果 |
|:---|:---|:---|:---:|
| 初始 | — | 空, 空 | — |
| j=0 | Block 0, Tag=0 | **[Tag0]**, 空 | **M** |
| j=2 | Block 2, Tag=1 | Tag0, **[Tag1]** | **M** |
| j=4 | Block 4, Tag=2 | **[Tag2]**, Tag1（踢 Tag0） | **M** |
| i=1, j=0 | Block 0, Tag=0 | Tag2, **[Tag0]**（踢 Tag1） | **M** |
| i=1, j=2 | Block 2, Tag=1 | **[Tag1]**, Tag0（踢 Tag2） | **M** |
| i=1, j=4 | Block 4, Tag=2 | Tag1, **[Tag2]**（踢 Tag0） | **M** |
| ... | ... | 反复踢 | **全是 M** |

Set 1 情况完全对称，同样全部 Miss。

##### C1) 不使用高速缓存

每次访问直接访问内存，$100\text{ ns}$/次：

$$\text{总时间} = 24 \times 100 = \mathbf{2400\text{ ns}}$$

##### C2) 使用高速缓存

**全部 24 次访问均为 Miss**（每次都要从内存加载整块，且加载后下一轮就被踢走）：

$$\text{总时间} = 24 \times 101 = \mathbf{2424\text{ ns}}$$

> ⚠️ 使用 Cache 后反而比不用 Cache 还慢（$2424 > 2400$），因为每次都是 Miss，白白多付了 24 次 Cache 查找的 1 ns 开销。这正是**按列访问 + 小容量 Cache** 的典型"抖动（thrashing）"现象——块被反复加载又踢出，Cache 形同虚设。

##### 对比 B 和 C

| | B（逐次访问指定元素） | C（按列循环） |
|:---|:---:|:---:|
| 访问次数 | 6 | 24 |
| 无 Cache | 600 ns | 2400 ns |
| 有 Cache | 306 ns | 2424 ns |
| 命中率 | 3/6 = 50% | 0/24 = 0% |
| Cache 效果 | ✅ 加速 ~2× | ❌ 反而变慢 |

**结论**：循环的访问顺序对 Cache 性能影响巨大。按行遍历（B 中隐含的行内连续访问）能利用空间局部性；按列遍历（C）在 Cache 容量不足以容纳所有行时会导致全部 Miss。
